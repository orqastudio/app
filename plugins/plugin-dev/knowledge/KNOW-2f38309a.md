# Plugin Development

## Environment Detection

Before starting any plugin work, determine the context automatically:

**First-party** (KNOW-e6fee7a0): The working directory is inside the platform dev environment — a repository with a `plugins/` directory and a project governance directory at root, and a submodule configuration referencing platform org repos. First-party plugins are submodules in the dev environment.

**Third-party** (KNOW-1b7fa054): Everything else. The plugin is a standalone project with its own project governance directory and the software plugin pre-installed for independent lifecycle management.

Load the appropriate sub-knowledge artifact based on detection. Do NOT ask the user — infer from the environment.

## Common Knowledge (Both Paths)

### Plugin Manifest (`orqa-plugin.json`)

Every plugin requires `orqa-plugin.json` at root:

```json
{
  "name": "@scope/plugin-name",
  "displayName": "Human Name",
  "version": "0.1.0-dev",
  "description": "One-line description.",
  "category": "coding-standards|delivery|integration|custom",
  "provides": {
    "knowledge": [],
    "schemas": [],
    "views": [],
    "relationships": [],
    "enforcement_mechanisms": [],
    "hooks": []
  },
  "content": {
    "knowledge": { "source": "knowledge", "target": ".orqa/process/knowledge" },
    "rules": { "source": "rules", "target": ".orqa/process/rules" }
  },
  "dependencies": {
    "npm": []
  },
  "build": "npm run build"
}
```

**`provides`** — declares what the plugin contributes to the engine (schemas, views, hooks, agents, knowledge, enforcement_mechanisms, etc.). At least one capability or a `content` mapping is required.

**`content`** — maps plugin source directories to `.orqa/` target paths. Files are copied at install time and tracked in `.orqa/manifest.json`. The engine scans `.orqa/`, not plugin source directories.

**`dependencies`** — npm packages installed into the project when this plugin is installed.

**`build`** — command to run before content is copied (e.g., to compile TypeScript).

See the Plugin Manifest Schema Reference documentation for the full field reference.

### Content Ownership and Manifest Tracking

When a plugin is installed, its content directories are copied to the project's `.orqa/` directory. The install is recorded in `.orqa/manifest.json`:

```json
{
  "plugins": {
    "@scope/plugin-name": {
      "version": "0.1.0-dev",
      "installed_at": "2026-03-23T00:35:00.000Z",
      "files": [
        ".orqa/process/knowledge/KNOW-abc123.md",
        ".orqa/process/rules/RULE-def456.md"
      ]
    }
  }
}
```

The engine uses `manifest.json` to:
- Know which files belong to which plugin
- Prevent direct edits to plugin-owned files
- Track files to remove on uninstall

### Edit Protection

Plugin-owned files in `.orqa/` cannot be edited directly. The engine rejects edits to any file listed in `manifest.json`. To update plugin content:

1. Edit the source file in the plugin directory (e.g., `plugins/my-plugin/knowledge/KNOW-abc123.md`)
2. Run `orqa plugin refresh` to rebuild and re-sync content to `.orqa/`

### Plugin Lifecycle Commands

```bash
orqa plugin install <name-or-path>   # Install a plugin and copy its content
orqa plugin uninstall <name>          # Remove plugin and delete its owned files
orqa plugin enable <name>             # Re-copy content for a disabled plugin
orqa plugin disable <name>            # Remove content without uninstalling
orqa plugin refresh [name]            # Rebuild and re-sync content (one or all)
orqa plugin diff [name]               # Show content drift between source and .orqa/
```

### Extension Pattern

Plugins can extend other plugins via the `extends` field:

```json
{
  "extends": ["@platform/plugin-rust"],
  "configExtensions": {
    "tsconfig": { "compilerOptions": { } },
    "eslint": { "import": "@scope/plugin/eslint", "export": "config" }
  }
}
```

### Templates

Four scaffolding templates are available:

| Template | Use When |
|----------|----------|
| `cli-tool` | One-shot CLI commands (build tools, generators, analysers) |
| `frontend` | Views and dashboards rendered in the platform explorer |
| `sidecar` | Long-running provider processes (AI services, language servers) |
| `full` | All of the above — views, sidecar, CLI tools, and hooks |

### Artifact Graph SDK

Plugins consume the SDK for graph queries:

```typescript
import { getStores } from "@platform/sdk";
const { artifactGraphSDK } = getStores();

// Resolution
artifactGraphSDK.resolve("EPIC-797972a7");
artifactGraphSDK.resolveByPath("path/to/file.md");

// Traversal
artifactGraphSDK.referencesFrom("EPIC-797972a7");
artifactGraphSDK.referencesTo("TASK-5ebf808b");

// Bulk queries
artifactGraphSDK.byType("task");
artifactGraphSDK.byStatus("active");

// Content
await artifactGraphSDK.readContent("path/to/file.md");

// Subscriptions (reactive updates)
artifactGraphSDK.subscribe("EPIC-797972a7", callback);
```

### Knowledge + Doc Requirement

Every plugin that defines artifact types or relationships MUST ship:
- A **knowledge artifact** teaching agents how to use the artifacts
- A **documentation** artifact teaching humans the same
- Connected via `synchronised-with` relationship

### Enforcement

Run `orqa enforce` in the plugin directory to check:
- Manifest schema compliance
- Knowledge/doc frontmatter validity
- Relationship target resolution
- ID uniqueness
- Template compatibility (for template repos)

## Anti-Patterns

| Anti-Pattern | Correct Approach |
|-------------|-----------------|
| Editing plugin-owned files directly in `.orqa/` | Edit in plugin source, then `orqa plugin refresh` |
| Developing inside production governance directory | Always scaffold from a template |
| Hardcoding artifact paths | Use SDK resolution methods |
| Polling for changes | Use SDK subscriptions |
| Skipping enforcement | Run `orqa enforce` before every commit |
| Missing knowledge+doc pair | Every artifact-providing plugin needs both |
| Missing `content` or `provides` | Every plugin must declare at least one capability |
