Software is built from small, composable units at every level: functions, modules, features, and integrations. Composable systems are dramatically easier to pivot, extend, and maintain.

## Core Idea

Every piece of the system should be: (1) small enough to understand in isolation, (2) pure enough to test without the world, (3) typed enough to compose safely, (4) swappable enough to replace without cascading changes.

## Seven Principles

| # | Principle | Key Rule |
|---|-----------|----------|
| 1 | Pure over stateful | Functions take inputs, return outputs. Isolate side effects at boundary layers. Domain logic stays pure. |
| 2 | Small composable units | Domain: 20-30 lines, service: 30-50, utility: 10-20. Top-level function reads like a table of contents. |
| 3 | Type boundaries | Define contracts via traits/interfaces. Consumers depend on the contract, not the implementation. Use discriminated unions for variants. |
| 4 | Pluggable by default | Every integration point is swappable. Design interface first, then implement. |
| 5 | Pipeline composition | Complex ops = chains of simple transforms. No intermediate mutable variables. |
| 6 | Feature as module | Self-contained: domain + service + types + state + UI. Dependency flow: UI -> state -> service -> domain. |
| 7 | Functional paradigm | Prefer map/filter/collect over mutation loops. Derived state over imperative updates. |

## Generic Composition Pattern

Design generic functions, then create typed entry points:
```
parse_document<T>(content) -> (T, body)   // generic core
parse_rule(content) -> (RuleFrontmatter, body)  // typed entry point
```

## Anti-Patterns

| Anti-Pattern | Fix |
|---|---|
| God components/functions | Split into smaller units with single responsibilities |
| Thick command handlers | Move logic into domain functions; handler becomes thin delegation |
| Circular dependencies | Extract shared logic into a third module |
| Tightly coupled features | Communicate through public interfaces, not internals |

## Composability Checklist

- Pure functions testable without external dependencies
- Every function within size guidelines and single-responsibility
- Interfaces defined by contracts, not concrete implementations
- External dependencies swappable without domain logic changes
- Complex operations as transform chains, not nested conditionals
- Features removable without breaking other features
- Unidirectional dependency flow (UI -> state -> service -> domain)
