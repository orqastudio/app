# Documentation Placement Guide

## Overview

OrqaStudio content lives in two kinds of locations, and the choice between them determines whether content ships with the product or stays local to the development project.

- **Plugin directories** (`plugins/<name>/`) — canonical source for production content
- **Project directory** (`.orqa/`) — development-only content plus installed copies of plugin content

This guide explains how to decide where new documentation and knowledge artifacts belong.

## The Plugin-Canonical Model

Plugins are the **canonical source of truth** for all content that ships with OrqaStudio. When you run `orqa install`, the CLI reads each plugin's manifest (`orqa-plugin.json`) and syncs content from the plugin's source directories into `.orqa/`:

```text
Plugin (canonical)              orqa install           .orqa/ (installed copy)
──────────────────              ────────────           ─────────────────────
plugins/core/knowledge/     ──────────────>  .orqa/documentation/knowledge/
plugins/core/rules/         ──────────────>  .orqa/learning/rules/
plugins/core/agents/        ──────────────>  .claude/agents/
plugins/core/docs/          ──────────────>  .orqa/documentation/
plugins/<name>/knowledge/   ──────────────>  .orqa/documentation/knowledge/
plugins/<name>/docs/        ──────────────>  .orqa/documentation/
```text

**Edit the plugin source, not the installed copy.** Changes to installed copies in `.orqa/` are overwritten on the next `orqa install`.

## Where Does My Content Belong?

### Decision Flowchart

```text
Is this content about a feature that ships with the product?
├── YES → Would it exist in a brand-new project after orqa install?
│         ├── YES → Write in the relevant plugin directory
│         │         ├── Framework-level (roles, delegation, search) → plugins/core/
│         │         ├── Domain-specific (Svelte patterns) → plugins/svelte/
│         │         ├── Technology-specific (Tauri IPC) → plugins/tauri/
│         │         └── Project-type defaults → plugins/software-kanban/
│         └── NO  → Write in .orqa/ (project-specific)
└── NO  → Is this about THIS project's development process?
          ├── YES → Write in .orqa/
          │         ├── Architecture decisions → .orqa/process/decisions/
          │         ├── Lessons learned → .orqa/learning/lessons/
          │         ├── Planning artifacts → .orqa/implementation/
          │         ├── Dev documentation → .orqa/documentation/
          │         └── Project-specific rules → .orqa/learning/rules/
          └── NO  → Reconsider whether it needs to be written at all
```text

### Plugin Directory Content

Write in a plugin when the content describes something that is part of the product:

| Content Type | Example | Plugin |
| ----------- | ------- | ------ |
| Core framework knowledge | Agent roles, delegation rules, search methodology | `plugins/core/` |
| Orchestrator and agent definitions | Universal roles, agent capabilities | `plugins/core/` |
| Domain-specific patterns | Svelte component patterns, runes usage | `plugins/svelte/` |
| Technology-specific guidance | Tauri IPC patterns, command structure | `plugins/tauri/` |
| CLI command documentation | CLI usage, commands, plugins | `plugins/cli/` |
| Project type templates | Software project defaults, coding standards | `plugins/software-kanban/` |

### Project Directory Content

Write in `.orqa/` when the content is specific to this project's development:

| Content Type | Example | Location |
| ----------- | ------- | -------- |
| Architecture decisions | Decisions about this project's design | `.orqa/process/decisions/` |
| Lessons learned | Mistakes and patterns discovered during development | `.orqa/learning/lessons/` |
| Project-specific rules | Rules unique to this project | `.orqa/learning/rules/` |
| Planning artifacts | Epics, tasks, ideas, research | `.orqa/implementation/` |
| Development documentation | Coding standards, workflow guides specific to this project | `.orqa/documentation/` |

## The Docs-Knowledge Pairing Rule

Every piece of documented content must exist as **two paired artifacts**:

1. **Documentation page** (`docs/` or `documentation/`) — user-facing. Written for human developers who want to understand a feature. Readable, contextual, narrative.
2. **Knowledge artifact** (`knowledge/`) — agent-facing. Written to be injected into agent context when working in a related code area. Structured, concise, action-oriented.

This pairing applies at both levels:

| Level | Documentation | Knowledge |
| ----- | ------------ | --------- |
| Plugin | `plugins/<name>/docs/` | `plugins/<name>/knowledge/` |
| Project | `.orqa/documentation/` | `.orqa/documentation/knowledge/` |

Link paired artifacts using the `synchronised-with` relationship type in their YAML frontmatter.

### Why Both Are Required

| Scenario | Problem |
| -------- | ------- |
| Knowledge exists, docs missing | Agents have context but humans cannot find explanations |
| Docs exist, knowledge missing | Humans have docs but agents lack context injection — agents make mistakes |
| Both exist and paired | Humans read the guide, agents get structured context injected automatically |

Creating one without the other is incomplete work.

## Editing Content

### Editing Plugin Content (Canonical)

1. Find the canonical source in `plugins/<name>/`
2. Edit the file in the plugin directory
3. Run `orqa install` to sync the change into `.orqa/`

**Do not edit the installed copy in `.orqa/` directly** — your changes will be overwritten.

### Editing Project Content (Dev-Only)

Edit directly in `.orqa/`. These files are not managed by `orqa install` and will not be overwritten.

### Adding a New Doc or Knowledge Artifact

1. Determine placement using the decision flowchart above
2. Create both the doc and knowledge artifact (pairing rule)
3. Add `synchronised-with` relationship linking them
4. If in a plugin: run `orqa install` to sync to `.orqa/`

## Drift Detection

OrqaStudio uses a three-way diff model to detect when content has drifted:

| Divergence | Meaning |
| ---------- | ------- |
| Project copy differs from installed baseline | A local edit was made to an installed copy |
| Plugin source differs from installed baseline | A plugin update changed the canonical source |

Both cases surface during `orqa install` for reconciliation.

## Common Mistakes

| Mistake | Correction |
| ------- | ---------- |
| Writing knowledge in `.orqa/documentation/knowledge/` for a product feature | Write in the plugin's `knowledge/` directory |
| Editing an installed copy in `.orqa/` directly | Edit the canonical source in the plugin, then `orqa install` |
| Creating a doc without a matching knowledge artifact | Always create both — they come in pairs |
| Creating knowledge without a matching doc page | Always create both — they come in pairs |
| Putting dev-only content in a plugin | Plugin content ships to all projects — dev content stays in `.orqa/` |
