# Phase 2: Proposed Codebase Restructure

**Date:** 2026-03-26
**Input:** ARCHITECTURE.md, Phase 1 inventories (01, 05, 06, 07, 09, 10)

---

## Design Goals

1. The folder structure makes the architecture self-evident to anyone reading a directory listing
2. The engine's independence from the app is visible at the top level
3. Plugin types are separated so it is hard to put a methodology plugin in the knowledge folder
4. Connectors, sidecars, and infrastructure plugins each have obvious homes
5. Shared libraries, infrastructure tooling, and templates have clear boundaries
6. Legacy/migration artifacts have no permanent home -- they must be cleaned up

---

## Proposed Top-Level Structure

```
orqastudio-dev/
  engine/                    # The standalone engine (Rust crates)
  app/                       # Desktop app: Tauri + SvelteKit (consumer of engine)
  plugins/
    methodology/             # One-at-a-time methodology definitions
    workflows/               # One-per-stage workflow definitions
    knowledge/               # Domain knowledge plugins
    infrastructure/          # Enforcement infrastructure (githooks, coding-standards)
    sidecars/                # LLM provider integrations for the app
  connectors/                # Generation plugins that produce tool-native plugins
  libs/                      # Shared libraries (TypeScript + Rust)
  infrastructure/            # Deployment infrastructure (git server, sync bridge)
  templates/                 # Scaffolding templates for new plugins
  scripts/                   # Build/install/migration scripts
  .orqa/                     # Governance artifacts (project-level)
  .claude/                   # Claude Code configuration (gitignored)
  .state/                    # Operational state (gitignored)
  .forgejo/                  # CI/CD workflows
```

## Directory Details

### 1. `engine/` -- The Standalone Engine

**Purpose:** The fixed core that everything composes around. Contains all business logic: graph, workflow, enforcement, prompt pipeline, search, agent generation. A standalone Rust workspace, not embedded in any consumer.

**Currently:** Engine business logic is split between:
- `libs/validation/` -- graph validation, integrity checks, relationship schemas
- `libs/search/` -- chunking, embedding, storage, search
- `libs/mcp-server/` -- MCP access protocol
- `libs/lsp-server/` -- LSP access protocol
- `app/src-tauri/src/domain/` -- enforcement engine (783 lines), status transitions (880 lines), process gates (548 lines), knowledge injector (477 lines), governance scanner (422 lines), hook manager (415 lines), project scanner (396 lines), artifact core (687 lines), artifact reader (874 lines), stream loop (1042 lines), tool executor (1140 lines), and more

**What moves in:**
- `libs/validation/` -> `engine/crates/validation/`
- `libs/search/` -> `engine/crates/search/`
- `libs/mcp-server/` -> `engine/crates/mcp-server/`
- `libs/lsp-server/` -> `engine/crates/lsp-server/`
- All `app/src-tauri/src/domain/` business logic modules (enforcement engine, status transitions, process gates, knowledge injector, governance scanner, artifact core, artifact reader, project scanner, etc.) -> `engine/crates/core/`

**What stays out:**
- Tauri glue code (commands, state, watcher, sidecar lifecycle, DB init, logging, startup tracker) stays in `app/`
- The app imports the engine as a workspace dependency

**Why this makes the architecture self-evident:**
- Looking at the top level, `engine/` is clearly the core -- separate from `app/`
- The engine has its own `Cargo.toml` workspace with its own crates
- No one would confuse where to put new business logic vs new UI features
- MCP and LSP are visibly inside the engine (access protocols, not separate apps)

**Proposed engine internal structure:**

```
engine/
  Cargo.toml                 # Workspace root for engine crates
  crates/
    core/                    # Graph, workflow, enforcement, prompt pipeline, agent generation
    validation/              # Artifact graph validation, integrity checks (from libs/validation)
    search/                  # ONNX embeddings, semantic search (from libs/search)
    mcp-server/              # MCP access protocol (from libs/mcp-server)
    lsp-server/              # LSP access protocol (from libs/lsp-server)
```

### 2. `app/` -- The Desktop Application

**Purpose:** UI consumer of the engine. Tauri (Rust backend as thin glue) + SvelteKit frontend. Contains only UI logic, Tauri command wiring, and sidecar lifecycle management.

**Currently:** `app/` contains both Tauri glue AND engine business logic mixed together in `src-tauri/src/domain/`. Also contains `.githooks/` with 16 validation scripts that duplicate engine enforcement logic.

**What moves out:**
- `app/src-tauri/src/domain/` business logic modules -> `engine/crates/core/`
- `app/.githooks/` validation scripts -> generated by `plugins/infrastructure/githooks/` from engine rules (target state)
- `app/tools/` migration/verification scripts -> `scripts/` (repo root) or deleted if one-off
- `app/scripts/rebuild-artifacts.mjs` -> `scripts/migration/` (one-off migration)
- `app/WORKING-DOCUMENT.md` -> archived or integrated into `ARCHITECTURE.md`

**What stays:**
- `app/src-tauri/` (Tauri glue: commands, state, watcher, sidecar, servers, db, logging)
- `app/src/` (SvelteKit frontend)
- `app/package.json`, `app/svelte.config.js`, `app/vite.config.ts`, etc.
- `app/src-tauri/Cargo.toml` (now depends on `engine` workspace crates)

**Why this makes the architecture self-evident:**
- `app/` is visibly just a UI consumer -- no one would look here for the enforcement engine
- The app's Cargo.toml imports engine crates, making the dependency explicit

### 3. `plugins/` -- Plugin Ecosystem (by type)

**Purpose:** All first-party plugins, organized by the plugin taxonomy from ARCHITECTURE.md section 4.1. The subdirectory names match the taxonomy, making it immediately clear what kind of plugin goes where.

**Currently:** All plugins are flat under `plugins/` with no type separation:
```
plugins/
  agile-workflow/       (methodology)
  agile-discovery/      (workflow)
  agile-planning/       (workflow)
  agile-documentation/  (workflow)
  agile-review/         (workflow)
  software-kanban/      (workflow + views)
  core/                 (workflow: learning stage)
  cli/                  (knowledge)
  rust/                 (knowledge + infrastructure)
  svelte/               (knowledge + infrastructure)
  tauri/                (knowledge)
  typescript/           (knowledge + infrastructure)
  coding-standards/     (infrastructure)
  systems-thinking/     (knowledge)
  plugin-dev/           (knowledge)
  githooks/             (infrastructure)
```

**Proposed structure:**

```
plugins/
  methodology/
    agile-workflow/              # The Agile methodology skeleton
  workflows/
    core/                        # Learning stage (the app's USP)
    agile-discovery/             # Discovery stage
    agile-planning/              # Planning stage
    agile-documentation/         # Documentation stage
    agile-review/                # Review stage
    software-kanban/             # Implementation stage (+ views)
  knowledge/
    cli/                         # CLI tool domain knowledge
    rust/                        # Rust development patterns
    svelte/                      # Svelte development patterns
    tauri/                       # Tauri development patterns
    typescript/                  # TypeScript configs and patterns
    systems-thinking/            # Systems design knowledge
    plugin-dev/                  # Plugin development knowledge
  infrastructure/
    githooks/                    # Git hooks from engine rules
    coding-standards/            # Code quality rules and tooling
  sidecars/
    (empty for now -- future LLM provider integrations)
```

| `plugins/core/` | `plugins/workflows/core/` |
| `plugins/agile-discovery/` | `plugins/workflows/agile-discovery/` |
| `plugins/agile-planning/` | `plugins/workflows/agile-planning/` |
| `plugins/agile-documentation/` | `plugins/workflows/agile-documentation/` |
| `plugins/agile-review/` | `plugins/workflows/agile-review/` |
| `plugins/software-kanban/` | `plugins/workflows/software-kanban/` |
| `plugins/cli/` | `plugins/knowledge/cli/` |
| `plugins/rust/` | `plugins/knowledge/rust/` |
| `plugins/svelte/` | `plugins/knowledge/svelte/` |
| `plugins/tauri/` | `plugins/knowledge/tauri/` |
| `plugins/typescript/` | `plugins/knowledge/typescript/` |
| `plugins/systems-thinking/` | `plugins/knowledge/systems-thinking/` |
| `plugins/plugin-dev/` | `plugins/knowledge/plugin-dev/` |
| `plugins/githooks/` | `plugins/infrastructure/githooks/` |
| `plugins/coding-standards/` | `plugins/infrastructure/coding-standards/` |

**Why this makes the architecture self-evident:**
- The folder names match the plugin taxonomy in ARCHITECTURE.md section 4.1
- You cannot accidentally put a methodology plugin in `plugins/knowledge/` -- the names enforce meaning
- The constraint "one methodology plugin at a time" is visible: `plugins/methodology/` has one entry
- The constraint "one workflow plugin per stage" maps to the `plugins/workflows/` directory
- Dual-purpose plugins (e.g., `rust` provides knowledge AND generates linting infrastructure) go in their primary category. The `orqa-plugin.json` manifest declares what each plugin provides -- the folder hints at the primary intent

**Note on dual-purpose plugins:** `rust`, `svelte`, and `typescript` all provide domain knowledge AND generate linting/enforcement infrastructure. They are categorized under `knowledge/` because their primary purpose is domain expertise. The infrastructure they generate (clippy configs, ESLint configs) is a secondary concern declared in their manifest. If this causes confusion, an alternative is to keep all plugins flat and rely solely on the `orqa-plugin.json` `category` field -- but the grouped structure makes the taxonomy more discoverable.

### 4. `connectors/` -- Generation Plugins

**Purpose:** OrqaStudio plugins that generate tool-native plugins for third-party tools. Currently just Claude Code; future connectors for Cursor, Copilot, etc.

**Currently:** `connectors/claude-code/` -- correct location, no change needed.

**What changes:**
- No structural change. This directory is already correctly scoped.
- The `integrations/claude-agent-sdk/` sidecar moves to `plugins/sidecars/claude-agent-sdk/`

**Migration map:**
| Current | Proposed |
|---------|----------|
| `connectors/claude-code/` | `connectors/claude-code/` (no change) |
| `integrations/claude-agent-sdk/` | `plugins/sidecars/claude-agent-sdk/` |

**Why this makes the architecture self-evident:**
- `connectors/` at the top level is a peer to `app/` -- both are interfaces to the engine
- It is clear that a connector is NOT a regular plugin -- it has its own top-level directory
- The `integrations/` directory is eliminated. "Integration" was ambiguous -- it is actually a sidecar plugin and belongs under `plugins/sidecars/`

### 5. `libs/` -- Shared Libraries

**Purpose:** Reusable TypeScript libraries shared across the app, connectors, and plugins.

**Currently:** `libs/` contains both TypeScript libraries AND Rust crates mixed together:
```
libs/
  types/              (TypeScript types -- shared)
  cli/                (TypeScript CLI -- shared)
  sdk/                (TypeScript SDK -- app + plugins)
  svelte-components/  (Svelte UI components -- app + plugins)
  graph-visualiser/   (Svelte graph viz -- app)
  brand/              (Design tokens + icon generation)
  logger/             (TypeScript logger)
  validation/         (Rust crate -- engine)
  search/             (Rust crate -- engine)
  mcp-server/         (Rust crate -- engine)
  lsp-server/         (Rust crate -- engine)
```

**What moves out:**
- `libs/validation/` -> `engine/crates/validation/`
- `libs/search/` -> `engine/crates/search/`
- `libs/mcp-server/` -> `engine/crates/mcp-server/`
- `libs/lsp-server/` -> `engine/crates/lsp-server/`

**What stays:**
- `libs/types/` -- TypeScript types shared across the ecosystem
- `libs/cli/` -- TypeScript CLI (`orqa` binary)
- `libs/sdk/` -- TypeScript SDK for app and plugin views
- `libs/svelte-components/` -- Shared Svelte UI components
- `libs/graph-visualiser/` -- Graph visualization component
- `libs/brand/` -- Design tokens, icon generation
- `libs/logger/` -- TypeScript logger

**Why this makes the architecture self-evident:**
- `libs/` is purely TypeScript -- no Rust/engine crates hiding here
- The engine's Rust crates live under `engine/crates/`, making the language boundary obvious
- The `libs/cli/` is the CLI binary -- it consumes the engine (spawns binaries, calls daemon API)

## Impact Assessment

### High Impact (Significant Work)

1. **Engine extraction from `app/src-tauri/src/domain/`** -- This is the largest structural change. ~8,000+ lines of business logic need to be extracted from the Tauri backend into `engine/crates/core/`. This requires:
   - Creating the new crate with proper module structure
   - Updating all `use` statements in the Tauri glue code
   - Ensuring the Tauri commands delegate to engine crate APIs
   - Updating Cargo.toml dependencies
   - Ensuring build order is correct

2. **Plugin reorganization** -- 16 plugins need to move to new directories. This requires:
   - Moving directories
   - Updating all npm workspace references
   - Updating `orqa-plugin.json` content mappings if they use relative paths
   - Updating any hardcoded plugin paths in the CLI or engine
   - Updating CI/CD workflows

### Medium Impact

3. **Script consolidation** -- Moving scripts between `app/scripts/`, `scripts/`, and `tools/`. Lower risk because these are standalone scripts with few dependencies.

4. **Rust workspace changes** -- Adding the new `engine/` workspace member and updating `app/src-tauri/Cargo.toml` to depend on it.

### Low Impact

5. **Removing legacy files** -- `validation_stderr.txt`, `tmp/`, `app/WORKING-DOCUMENT.md`. Safe to delete.

6. **Moving `integrations/` to `plugins/sidecars/`** -- The claude-agent-sdk has no dependents currently (empty hooks, stub sidecar).

---

## Sequencing Recommendation

The restructure should be done in phases to avoid a single massive breaking change:

**Phase 1: Engine extraction** (highest value, highest risk)
- Create `engine/crates/core/` and move business logic from `app/src-tauri/src/domain/`
- Move `libs/validation/`, `libs/search/`, `libs/mcp-server/`, `libs/lsp-server/` to `engine/crates/`
- Update Cargo workspace
- Verify builds and tests pass

**Phase 2: Plugin reorganization** (medium value, medium risk)
- Create subdirectory structure under `plugins/`
- Move plugins to their type-specific directories
- Update npm workspaces
- Update any hardcoded paths in CLI or engine
- Move `integrations/claude-agent-sdk/` to `plugins/sidecars/`
- Remove `integrations/` directory

**Phase 3: Script and tools consolidation** (low value, low risk)
- Create `scripts/migration/` and move migration scripts
- Create `tools/verify/` and move verification scripts
- Clean up `app/scripts/` and `app/tools/`
- Remove legacy files

**Phase 4: Githooks migration** (depends on engine enforcement pipeline)
- Replace `app/.githooks/` with generated hooks from `plugins/infrastructure/githooks/`
- This depends on the enforcement pipeline being mature enough to generate equivalent hooks

## Validation Criteria

After the restructure:

1. `engine/` is a standalone Rust workspace that compiles independently of `app/`
2. `app/src-tauri/` contains only Tauri glue code -- no business logic modules
3. All 16 plugins are under their type-specific directory in `plugins/`
4. `libs/` contains only TypeScript packages -- no Rust crates
5. `npm install` and `cargo build` both succeed from the repo root
6. All existing tests pass
7. `orqa install` works correctly with the new plugin paths
8. The connector can find and generate plugins from the new structure
9. CI/CD workflows pass
