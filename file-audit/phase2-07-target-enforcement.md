# Phase 2-07: Target Enforcement Tooling Stack

## Question

What should the COMPLETE enforcement tooling stack look like? Define the target state for git hooks, linting configs, artifact validation rules, agent permission scopes, and workflow guards -- as test fixtures that the generation pipeline must produce.

---

## 1. Git Hooks (Target)

ARCHITECTURE.md Section 10.1 states: git hooks are "generated by plugins from engine rules." The target state has two layers: the **githooks plugin** (thin CLI wrapper) and the **app/.githooks/** directory (project-specific hooks generated at install time). The plugin hook is the canonical one; the app hooks are legacy and should converge.

### 1.1 Pre-Commit Hook

**Purpose:** Block commits that violate mechanical quality standards. Thin wrapper around `orqa check` -- the hook itself holds zero business logic.

**Target checks (in order):**

| # | Check | Trigger Condition | Mechanism | Blocking |
|---|-------|-------------------|-----------|----------|
| 1 | Rust format | `.rs` or `Cargo.toml` staged | `orqa check rustfmt` | Yes |
| 2 | Rust lint | `.rs` or `Cargo.toml` staged | `orqa check clippy` | Yes |
| 3 | Frontend lint | `.svelte`, `.ts`, `.js` staged | `orqa check eslint` (staged files only) | Yes |
| 4 | Artifact validation | `.orqa/**/*.md` staged | `orqa check validate` | Yes |
| 5 | Relationship validation | `.orqa/**/*.md` staged | Part of `orqa check validate` | Yes |
| 6 | Status transition validation | `.orqa/**/*.md` staged | Part of `orqa check validate` | Yes |
| 7 | Plugin source protection | `.orqa/**` staged | Part of `orqa check validate` | Yes (dogfood: warn) |

**What is NOT in hooks (and should not be):**

- **Tests** (`cargo test`, `vitest run`, `svelte-check`): Too slow for pre-commit. Run in CI or on-demand via `orqa test`.
- **Pipeline integrity**: Business logic, not a commit-gate. Run via `orqa check verify` or in CI.
- **Pillar alignment**: Governance policy, enforced by the engine at artifact creation time, not at commit time.
- **Core graph protection**: Install-time protection via manifest, not commit-time shell scripting.
- **Epic readiness / task dependency gates**: Workflow guard logic belongs in the engine's state machine, not shell scripts.
- **Stub scanning**: Advisory, run in CI. Should not block commits.
- **Lint suppression audit**: Advisory, run in CI.
- **Auto-linking**: Side effect in a hook is an anti-pattern. Should be an explicit `orqa link` command.
- **Rule overrides / collect-rule-overrides**: Complex shell orchestration that belongs in the engine.

**Target pre-commit script (canonical form):**

```bash
#!/usr/bin/env bash
# Generated by @orqastudio/plugin-githooks
# Thin wrapper: delegates ALL logic to `orqa check`
set -euo pipefail

STAGED=$(git diff --cached --name-only --diff-filter=ACMR 2>/dev/null || true)
[ -z "$STAGED" ] && exit 0

if ! command -v orqa &>/dev/null; then
  echo "WARNING: orqa CLI not available -- skipping pre-commit checks"
  exit 0
fi

FAILED=false

# Rust checks (format + lint)
echo "$STAGED" | grep -qE '\.(rs)$|Cargo\.toml$' && {
  orqa check rustfmt 2>&1 || FAILED=true
  orqa check clippy  2>&1 || FAILED=true
}

# Frontend checks (lint only -- staged files)
FE_FILES=$(echo "$STAGED" | grep -E '\.(svelte|ts|js)$' || true)
[ -n "$FE_FILES" ] && {
  orqa check eslint --files "$FE_FILES" 2>&1 || FAILED=true
}

# Artifact validation (schema + relationships + transitions)
echo "$STAGED" | grep -qE '\.orqa/.*\.md$' && {
  orqa check validate 2>&1 || FAILED=true
}

[ "$FAILED" = true ] && { echo "BLOCKED: Fix errors before committing."; exit 1; }
echo "Pre-commit: All checks passed."
```

**Key differences from current state:**

- The plugin hook (`plugins/githooks/hooks/pre-commit`) is close to target -- it already delegates to `orqa check`. Keep this as the canonical hook.
- The app hook (`app/.githooks/pre-commit`, 160 lines) has 15+ inline validation scripts -- these must be consolidated into `orqa check validate`.
- Violation logging to `.state/precommit-violations.jsonl` is a good pattern -- keep it in the CLI check command, not in the hook shell script.

### 1.2 Post-Commit Hook

**Purpose:** Auto-push to remote after every commit (RULE-f609242f enforcement).

**Target behavior:**
- Skip during rebase, merge, cherry-pick, revert, detached HEAD
- Skip if no remote configured
- Set upstream on first push for new branches
- Warn on failure but never block

The current plugin post-commit hook is already at target state. No changes needed.

### 1.3 Pre-Action Hook (Agent Tool Interception)

**Purpose:** Validate relationship types before agent Write/Edit tool calls modify `.orqa/` artifacts.

**Target behavior:**
- Receives JSON on stdin from the connector's tool interception
- Delegates validation to the daemon's `/validate` endpoint
- Exit 0 = allow, Exit 2 = block with reason on stderr

The current `validate-relationships.mjs` is close to target. The only change: it should validate ALL artifact fields (not just relationships) by delegating to the same `orqa check validate` path.

### 1.4 Hook Installation

**Target:** `orqa install` sets `git config core.hooksPath` to the installed hooks directory. The hooks are copies (not symlinks) from the githooks plugin, placed at a predictable path.

### 2.7 Config Generation Pipeline

**Target flow:**
1. Rules in `.orqa/rules/` declare `enforcement` entries with `tool` references
2. `ConfigGenerator` reads rules, discovers plugin tools, groups enforcement entries by tool
3. Generates tool-specific config files (JSON, TOML) at the project root
4. `orqa check configure` triggers regeneration
5. Git hooks validate that config files match generated state (existing `validate-config-disk.mjs` pattern)

**Current status:** `ConfigGenerator` class exists in `plugins/coding-standards/src/config-generator.ts`. It reads rules from `.orqa/process/rules/`, discovers plugin tools, and generates configs. The pipeline is functional but the rules path needs updating to `.orqa/rules/` (target structure removes the `process/` nesting).

## 3. Artifact Validation Rules (Target)

These rules define what `orqa check validate` enforces. The validation engine lives in `libs/validation` (Rust crate) and is exposed via the daemon and CLI.

### 3.1 Required Frontmatter Fields Per Artifact Type

**Universal required fields (all artifact types):**

| Field | Format | Validation |
|-------|--------|------------|
| `id` | `PREFIX-[0-9a-f]{8}` | Must match filename, prefix must match type |
| `type` | string | Must be a registered artifact type from plugin schemas |
| `title` | string | Non-empty, max 120 characters |
| `status` | string | Must be from workflow-defined values |

**Type-specific required fields (defined by methodology/workflow plugins):**

| Artifact Type | Additional Required Fields |
|---------------|---------------------------|
| TASK | `description` |
| EPIC | `description` |
| MILESTONE | `description` |
| RULE | `description` |
| KNOWLEDGE | `description`, `summary` |
| DECISION | `description` |
| LESSON | `description`, `category` |
| RESEARCH | `description` |
| AGENT | `description`, `role` |

**Optional but validated when present:**

| Field | Validation |
|-------|------------|
| `relationships` | Array of `{target, type}` objects; both fields required |
| `priority` | One of: P0, P1, P2, P3 |
| `acceptance_criteria` | Array of `{text, met}` objects |
| `gate_questions` | Array of `{text, met}` objects |
| `enforcement` | Array of enforcement entry objects |

### 3.2 ID Format Requirements

| Rule | Pattern | Example |
|------|---------|---------|
| Format | `TYPE-[0-9a-f]{8}` | `TASK-abcd1234` |
| Prefix matches type | Type `task` -> prefix `TASK` | Required |
| Unique across project | No duplicate IDs | Graph-level check |
| Filename matches ID | File `TASK-abcd1234.md` has `id: TASK-abcd1234` | Required |

**Valid prefixes (from plugin schemas, NOT hardcoded):**
The validation engine reads registered artifact types from the composed schema. Each type declares its `id_prefix`. The engine does NOT maintain a hardcoded list of prefixes.

### 3.3 Type Must Match Directory Location

| Directory | Expected Type(s) |
|-----------|-------------------|
| `.orqa/delivery/epics/` | `epic` |
| `.orqa/delivery/tasks/` | `task` |
| `.orqa/delivery/milestones/` | `milestone` |
| `.orqa/delivery/ideas/` | `idea` |
| `.orqa/discovery/research/` | `research` |
| `.orqa/discovery/ideas/` | `idea` |
| `.orqa/principles/vision/` | `vision` |
| `.orqa/principles/pillars/` | `pillar` |
| `.orqa/principles/personas/` | `persona` |
| `.orqa/decisions/` | `decision` |
| `.orqa/knowledge/` | `knowledge` |
| `.orqa/rules/` | `rule` |
| `.orqa/documentation/` | `documentation` |
| `.orqa/lessons/` | `lesson` |

The mapping is defined in the composed schema, not hardcoded in the engine.

### 3.4 Relationship Validity

| Check | Description |
|-------|-------------|
| Target exists | `target` ID resolves to an existing artifact in the graph |
| Type is valid | `type` is in the registered relationship vocabulary (from plugin schemas) |
| From/To constraints | The relationship type's declared `from` and `to` artifact types include the source and target types |
| No self-references | Target is not the same artifact |
| No duplicate entries | Same `{target, type}` pair appears at most once |
| Forward-only storage | No inverse relationship types stored (graph computes inverses) |

**Valid relationship types come from:**
1. `core.json` platform relationships (currently empty -- plugins are the source)
2. Plugin manifest `provides.relationships` declarations
3. Project-level `project.json` custom relationships

### 3.5 Status Validity

| Check | Description |
|-------|-------------|
| Status is defined | Must be from the project's registered status definitions |
| Transitions are legal | Old status -> new status must be an allowed transition per the status machine |

Status definitions and transitions come from `project.json` settings, which are composed from plugin workflow definitions at install time.

### 3.6 Size Constraints

| Artifact Type | Token Range | Rationale |
|---------------|-------------|-----------|
| Knowledge | 500-2,000 tokens | ARCHITECTURE.md Section 12 criterion 5 |

Other artifact types have no enforced size constraints.

### 3.7 Summary of Validation Engine Responsibilities

The validation engine (`libs/validation`) should enforce all of the above. Currently it handles:
- Frontmatter parsing and field validation
- Relationship validation (types, targets)
- Graph integrity checks (orphans, broken refs, cycles)

It should additionally enforce (currently done in shell scripts):
- Status transition validation (currently `validate-status-transitions.mjs`)
- Type-location consistency (currently not mechanically enforced)
- ID format validation (currently partial)
- Plugin source protection (currently `validate-plugin-sources.mjs`)

## 4. Agent Permission Scopes (Target)

Per ARCHITECTURE.md Section 6.1, there are 5 base roles. Permission scopes define what files/directories each role can access. These are enforced by the generated tool-native plugin (e.g., Claude Code Plugin).

### 4.1 Orchestrator

| Permission | Scope | Rationale |
|------------|-------|-----------|
| Read files | All project files | Needs full visibility to coordinate |
| Write files | None | Orchestrator coordinates, does not implement |
| Shell commands | None | No direct execution |
| Create artifacts | `.state/team/` task findings only | Team coordination files |
| Special | Can spawn agents via team infrastructure | Hub-spoke orchestration |

### 4.2 Implementer

| Permission | Scope | Rationale |
|------------|-------|-----------|
| Read files | All project files | Needs codebase visibility |
| Write files | `app/src/**`, `app/src-tauri/src/**`, `libs/*/src/**`, `plugins/*/src/**`, test files | Source code only |
| Shell commands | Build, test, lint commands | Compile, run tests |
| Cannot write | `.orqa/**`, `CLAUDE.md`, config files | Governance is not code |
| Cannot write | `plugins/*/knowledge/**`, `plugins/*/rules/**` | Plugin content is authoritative |

### 4.3 Reviewer

| Permission | Scope | Rationale |
|------------|-------|-----------|
| Read files | All project files | Full visibility for review |
| Write files | None | Read-only role |
| Shell commands | `orqa check`, `orqa test`, `cargo test`, `npx vitest`, `npx eslint`, `npx svelte-check` | Can run checks but not modify |
| Special | Can create findings/verdict artifacts | Review output |

### 4.4 Researcher

| Permission | Scope | Rationale |
|------------|-------|-----------|
| Read files | All project files | Needs full visibility for investigation |
| Write files | Research artifacts only (`.orqa/discovery/research/`, `.state/team/*/task-*.md`) | Creates research output |
| Shell commands | None | No execution |
| Special | Can use semantic search, web search | Research tools |

### 4.5 Writer

| Permission | Scope | Rationale |
|------------|-------|-----------|
| Read files | All project files | Needs context for documentation |
| Write files | `.orqa/documentation/**`, `docs/**`, `*.md` (non-governance) | Documentation only |
| Shell commands | None | No execution |
| Cannot write | Source code, configs, governance artifacts | Scope limited to docs |

### 4.6 Additional Roles (from CLAUDE.md)

| Role | Write Scope | Shell | Notes |
|------|-------------|-------|-------|
| **Governance Steward** | `.orqa/` artifacts only | No | Governance artifact management |
| **Planner** | Delivery artifacts (`.orqa/delivery/`) | No | Epic/task/milestone management |
| **Installer** | Config files, `.orqa/manifest.json` | Yes (build/install) | Plugin installation |
| **Plugin Developer** | Plugin source code and manifests | Yes (build/test) | Plugin development |
| **Designer** | Design artifacts, wireframes | No | Design output |

### 4.7 Permission Enforcement Mechanism

Permissions are translated into the tool-native plugin format at install time by the connector:

- **Claude Code:** Permission scopes become `allow`/`deny` patterns in the generated `.claude-plugin/` plugin's configuration
- **Future connectors:** Same permission model, different output format

The engine holds the canonical permission definitions. The connector generates the enforcement artifact.

## 5. Workflow Guards (Target)

Workflow guards are mechanical checks that gate workflow transitions. They are evaluated by the engine's state machine, not by shell scripts.

### 5.1 Task Completion Guard

**Trigger:** Task status transition to `done` or `completed`.

**Mechanical checks:**

| Check | Description | How to verify |
|-------|-------------|---------------|
| Deliverable exists | Task `delivers` an artifact that exists in the graph | Graph query: outgoing `delivers` relationships resolve |
| Test exists | If task delivers code, a test file exists for the deliverable | File system check: corresponding `*.test.ts` or `#[cfg(test)]` module |
| Test passes | The test for the deliverable passes | `orqa test` execution on the specific test |
| AC verified | All acceptance criteria have `met: true` | Frontmatter field check |

### 5.2 Epic Completion Guard

**Trigger:** Epic status transition to `done` or `completed`.

**Mechanical checks:**

| Check | Description | How to verify |
|-------|-------------|---------------|
| All tasks completed | Every task with `delivers` relationship to this epic has status `done` or `completed` | Graph query: all children by `delivers` inverse have terminal status |
| AC verified | All acceptance criteria have `met: true` | Frontmatter field check |
| No blocked tasks | No task delivering to this epic has status `blocked` | Graph query |

### 5.3 Milestone Completion Guard

**Trigger:** Milestone status transition to `done` or `completed`.

**Mechanical checks:**

| Check | Description | How to verify |
|-------|-------------|---------------|
| All epics completed | Every epic with `delivers` relationship to this milestone has status `done` | Graph query |
| No in-progress epics | No epic delivering to this milestone has status `in-progress` | Graph query |

### 5.4 Status Transition Guards (General)

The status transition engine (`domain/status_transitions.rs`) already supports 5 condition types:

| Condition | Description | Example |
|-----------|-------------|---------|
| `all_children_status` | All children must have specified statuses | All tasks done before epic done |
| `any_child_status` | At least one child has specified status | At least one task started before epic in-progress |
| `no_children_status` | No children have specified status | No blocked tasks before epic done |
| `parent_status` | Parent must have specified status | Epic must be in-progress before task starts |
| `relationship_count` | Relationship count meets threshold | At least one `delivers` relationship |

These conditions are declared in plugin-provided relationship definitions (`statusRules`), making them fully config-driven. This is already at target architecture.

### 5.5 Process Gates (Session-Level)

The process gates engine (`domain/process_gates.rs`) provides 5 session-level gates:

| Gate | Purpose | Condition |
|------|---------|-----------|
| `understand-first` | Read before write | Files read count > 0 before first write |
| `docs-before-code` | Consult docs before implementing | Docs consulted > 0 before code write |
| `plan-before-build` | Have a plan before building | Plan artifact exists or search performed |
| `evidence-before-done` | Verify before marking done | Tests run, checks passed |
| `learn-after-doing` | Capture lessons after completion | Lesson artifact created |

These are evaluated during streaming sessions and influence tool approval decisions. They should remain in the engine, not in hooks.

### 5.6 Review Gates

**Trigger:** Artifact or PR entering review state.

**Mechanical checks before human review:**

| Check | Type | Description |
|-------|------|-------------|
| CI green | Automated | All `orqa check` commands pass |
| Coverage threshold | Automated | Test coverage meets 80% (from vitest config) |
| No blocking violations | Automated | No enforcement rule violations with `action: block` |
| Lint clean | Automated | ESLint and Clippy report zero errors |

These should be orchestrated by `orqa check` and surfaced in the UI as a review readiness indicator, not as additional hook scripts.

## 7. Enforcement Layer Architecture (Target Summary)

```
Layer 1: Composed JSON Schema
  Source: Plugin schemas composed at install time
  Enforces: Artifact structure, required fields, valid types, valid relationships
  Mechanism: LSP validation against schema.composed.json
  Scope: Real-time editing + pre-commit

Layer 2: Git Hooks (Generated)
  Source: githooks plugin, configured by engine rules
  Enforces: Pre-commit quality gate (lint + validate), post-commit push
  Mechanism: Thin bash wrapper around `orqa check`
  Scope: Commit time

Layer 3: Linting Configs (Generated)
  Source: Coding-standards rules -> ConfigGenerator -> tool configs
  Enforces: Code style (eslint, clippy, rustfmt)
  Mechanism: Standard tool execution via `orqa check <tool>`
  Scope: Pre-commit + CI + editor

Layer 4: Engine Validation (Runtime)
  Source: libs/validation crate + daemon
  Enforces: Frontmatter, IDs, types, relationships, transitions, sizes
  Mechanism: orqa check validate + MCP/LSP endpoints
  Scope: Pre-commit + real-time + API

Layer 5: Agent Permissions (Generated)
  Source: Methodology plugin role definitions
  Enforces: File access, artifact scope, tool constraints per role
  Mechanism: Connector generates tool-native plugin configs
  Scope: Agent runtime

Layer 6: Workflow Guards (Engine)
  Source: Plugin workflow definitions (statusRules)
  Enforces: Transition prerequisites, gate requirements, AC verification
  Mechanism: Engine state machine evaluation
  Scope: Artifact status changes
```

## Recommendations

1. **Consolidate app/.githooks/ into the engine.** The 15 shell-script validators in `app/.githooks/` must be absorbed into `libs/validation` and exposed via `orqa check validate`. The plugin hook already delegates correctly -- the app hooks are the gap.

2. **Make the plugin hook canonical.** Set `core.hooksPath` to the installed copy of the githooks plugin hooks. Remove `app/.githooks/` entirely once validation logic is in the engine.

3. **Complete the config generation pipeline.** The `ConfigGenerator` skeleton exists. Wire it to the actual rules path (`.orqa/rules/`, not `.orqa/process/rules/`), and have `orqa install` trigger config generation.
