## Governing Constraint

The ONLY acceptable outcome is COMPLETE adherence to the architecture defined in RES-d6e8ab11. ANY variance from the research MUST be raised as a decision request (AD artifact) for user review BEFORE implementation. Agents may NOT independently decide to deviate, defer, or reinterpret the architecture.

## Architecture Clarifications (2026-03-25)

### Install-time vs Runtime boundary

The `workflow-resolver.ts`, `prompt-registry.ts`, and `agent-file-generator.ts` all run at install time (called from `cmdPluginSync` in `install.ts`). They scan `plugins/`, `connectors/`, `integrations/` to compose content into `.orqa/`. This is architecturally correct — composition happens at install time (P7, RES-d6e8ab11 section 3). The violation boundary is narrow: only runtime code (daemon, connector hooks, MCP server) must read from `.orqa/` exclusively. Install-time code reading plugin source directories is the correct design.

### Plugin decomposition model

The research's three-layer plugin model maps to the existing plugin structure as follows:

- **Workflow-definition plugin** = `software-kanban` (renamed from `software`) — owns the delivery workflow skeleton with contribution points
- **Stage-definition plugins** = `agile-planning` (fills `planning-methodology`), `agile-review` (fills `review-process`), `software-kanban` itself (fills `implementation-workflow`)
- **Governance plugin** = `agile-governance` — scoped to methodology-agnostic governance primitives only (standalone workflows, schemas, governance rules/knowledge, universal role definitions)
- **Core plugin** = `core` — unchanged (agent execution model, universal roles, session management, learning-pipeline contribution)

### Connector architecture

The connector is a THIN adapter layer. NO business logic in the connector. All enforcement lives in the daemon via POST /hook. The connector must require daemon running (daemon-gate exists), and should be designed to require MCP and LSP servers (validated against code quality standards during migration, not against running services).

## Current Violations (identified 2026-03-25)

### V1: Runtime reads plugin source dirs instead of .orqa/ (NARROWER than originally assessed)

Install-time tools (`workflow-resolver.ts`, `prompt-registry.ts`, `agent-file-generator.ts`) reading `plugins/` is CORRECT — they compose content into `.orqa/` at install time. The violation is limited to any runtime code (daemon, connector hooks, MCP server) that reads `plugins/` directly instead of `.orqa/`. Validation must confirm this boundary is maintained.

### V2: Monolithic agile-governance plugin

Single plugin provides: standalone workflows, artifact type schemas, state machines, knowledge, rules, agent definitions, relationship types, AND the delivery workflow skeleton with contributions. The delivery skeleton and stage-filling contributions should be in separate plugins per the three-layer model (RES-d6e8ab11 section 3).

### V3: Software plugin naming and scope mismatch

`plugins/software/` is named `@orqastudio/plugin-software-project` with role `core:delivery`. It fills three contribution points (`implementation-workflow`, `planning-methodology`, `review-process`) but should only fill `implementation-workflow`. Planning and review are methodology concerns that belong in separate stage-definition plugins. The delivery workflow skeleton belongs in the workflow-definition plugin, not in `agile-governance`.

### V4: Connector may contain business logic

Connector hooks in `connectors/claude-code/src/hooks/` must be thin adapters: stdin → HookContext → daemon POST /hook → output. Any hook that evaluates enforcement rules, checks field values, or makes governance decisions locally instead of delegating to the daemon is a violation. The daemon is the business logic boundary (AD-1ef9f57c question 8).

### V5: Agent prompts not fully generated from pipeline

`.claude/agents/` files contain hardcoded content that should be pipeline output. Knowledge references list IDs without titles. No stage context (Layer 2) included. Pipeline should produce three-layer composition: universal role + stage context + domain knowledge (RES-d6e8ab11 section 4).

### V6: Required contribution points filled incorrectly

The delivery workflow has 3 required contribution points. Currently the software plugin fills all three, but planning and review should come from separate stage-definition plugins. Core learning-pipeline contribution is missing.

### V7: Rule enforcement not through daemon for all entry types

Rules don't declare entries for all daemon enforcement types (`field-check`, `tool-matcher`). Hand-written scripts duplicate daemon logic. Daemon is the business logic boundary (AD-1ef9f57c question 8).
