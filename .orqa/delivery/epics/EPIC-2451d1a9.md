## Governing Constraint

The ONLY acceptable outcome is COMPLETE adherence to the architecture defined in RES-d6e8ab11. ANY variance from the research MUST be raised as a decision request (AD artifact) for user review BEFORE implementation. Agents may NOT independently decide to deviate, defer, or reinterpret the architecture.

## Current Violations (identified 2026-03-25)

### V1: Runtime reads plugin source dirs instead of .orqa/
- `workflow-resolver.ts` scans `plugins/*/workflows/`
- `prompt-registry.ts` scans `plugins/*/orqa-plugin.json`
- `agent-file-generator.ts` reads pipeline which reads plugins
- Validation scans plugin source for broken link checks
- **Research says:** "The runtime reads only the resolved file" (P7). Install-time merging, runtime reads `.orqa/`.

### V2: Monolithic agile-governance plugin
- Single plugin provides: artifact type schemas, state machines, workflows, knowledge, rules, agent definitions, relationship types
- **Research says:** Three plugin types — workflow-definition, stage-definition, knowledge (Section 3). "A workflow-definition plugin defines the skeleton and stage slots, stage-definition plugins fill each slot" (line 157).

### V3: No install boundary
- `orqa install` doesn't fully sync plugin content into `.orqa/`
- Content file paths in manifests use `../../.orqa/` (backwards from plugin dirs)
- No three-way diff (plugin source vs installed vs customized)
- **Research says:** "Composition happens at install time, not runtime" (line 111). "Re-resolution happens only on plugin install/update" (line 147).

### V4: Connector contains business logic
- `enforce-background-agents.mjs` — field-check logic belongs in daemon
- `enforce-completion-gate.mjs` — completion logic belongs in daemon
- **Research says:** "Daemon runs business logic" (AD-1ef9f57c question 8). Connector hooks are thin adapters.

### V5: Agent prompts not generated from pipeline
- `.claude/agents/` files contain hardcoded content that should be pipeline output
- Knowledge references list IDs without titles
- No stage context (Layer 2) included
- **Research says:** "System prompts are generated programmatically from plugin registries and workflow state" (P3, line 87).

### V6: Required contribution points unfilled
- Delivery workflow has 3 required points incorrectly filled by software plugin
- Core learning-pipeline contribution missing
- **Research says:** Stage-definition plugins fill slots, not domain plugins (Section 3).

### V7: Rule enforcement not through daemon for all entry types
- `field-check` and `tool-matcher` added to daemon but rules don't declare entries for them
- Hand-written scripts duplicate daemon logic
- **Research says:** Daemon is business logic boundary (AD-1ef9f57c).
