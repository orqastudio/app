## Architecture Clarifications (2026-03-25)

### Install-time vs Runtime boundary

The `workflow-resolver.ts`, `prompt-registry.ts`, and `agent-file-generator.ts` all run at install time (called from `cmdPluginSync` in `install.ts`). They scan `plugins/`, `connectors/`, `integrations/` to compose content into `.orqa/`. This is architecturally correct — composition happens at install time (P7, RES-d6e8ab11 section 3). The violation boundary is narrow: only runtime code (daemon, connector hooks, MCP server) must read from `.orqa/` exclusively. Install-time code reading plugin source directories is the correct design.

### Plugin decomposition model

**Definitions vs Content:** Plugins provide artifact type DEFINITIONS (schemas, workflows, relationship types) and CONTENT (docs, knowledge, rules about their domain). A plugin dependency is created when a plugin has content that requires a definition from another plugin. Relationship definitions belong with their artifact definitions and declare cross-stage joins in the manifest, resolved at install time. Defaults assume companion plugins for the current environment (Agile workflows + software projects).

**Stage-scoped artifact types:** Concepts like idea, research, and decision exist in multiple stages but with different semantics. Each stage plugin defines its own types (e.g., `discovery-idea` vs `planning-idea`). Same concept, different lifecycle and context.

**Plugin artifact DEFINITIONS:**

| Plugin | Role | Artifact Definitions | Fills Slot |
|--------|------|---------------------|------------|
| `core` | `core:framework` | knowledge, rule, lesson, agent, doc | `learning-pipeline` |
| `agile-workflow` | `core:workflow` | *(none — skeleton only)* | *(defines the skeleton)* |
| `agile-discovery` | *(stage)* | vision, persona, pillar, pivot, wireframe, discovery-idea, discovery-research, discovery-decision | `discovery-artifacts` |
| `agile-planning` | *(stage)* | planning-idea, planning-research, planning-decision | `planning-methodology` |
| `software-kanban` | *(stage)* | epic, task, milestone | `implementation-workflow` |
| `agile-review` | *(stage)* | *(TBD — review-specific types)* | `review-process` |

**Plugin CONTENT** (docs, knowledge, rules about their domain) lives in the plugin with domain expertise. Domain plugins (rust, svelte, tauri, typescript, etc.) provide content requiring definitions from core or stage plugins. Project-specific artifacts exist only at project level.

**Workflow-definition plugin** = `agile-workflow` (renamed from `agile-governance`). Owns ONLY the agile-methodology workflow skeleton with contribution point slots (discover, plan, document, implement, review, learn). No artifact definitions. Governance is an architectural property, not a plugin concern.

**Core plugin** = `core`. Platform primitives (knowledge, rule, lesson, agent, doc definitions) + fills `learning-pipeline` (the one stage used consistently across all workflow variants).

### Connector architecture

The connector is a THIN adapter layer. NO business logic in the connector. All enforcement lives in the daemon via POST /hook. The connector must require daemon running (daemon-gate exists), and should be designed to require MCP and LSP servers (validated against code quality standards during migration, not against running services).

## Current Violations (identified 2026-03-25)

### V1: Runtime reads plugin source dirs instead of .orqa/ (NARROWER than originally assessed)

Install-time tools (`workflow-resolver.ts`, `prompt-registry.ts`, `agent-file-generator.ts`) reading `plugins/` is CORRECT — they compose content into `.orqa/` at install time. The violation is limited to any runtime code (daemon, connector hooks, MCP server) that reads `plugins/` directly instead of `.orqa/`. Validation must confirm this boundary is maintained.

### V2: Monolithic agile-workflow plugin

Single plugin provides: the delivery workflow skeleton, standalone workflows, artifact type schemas, state machines, knowledge, rules, agent definitions, relationship types, AND stage-filling contributions. Stage-filling contributions should be in separate stage-definition plugins, and discovery artifacts (idea) should be in a discovery plugin — per the three-layer model (RES-d6e8ab11 section 3). The skeleton stays in the workflow-definition plugin (agile-workflow) but must be renamed from `delivery` to `agile-methodology`.

### V3: Software plugin naming and scope mismatch

`plugins/software/` is named `@orqastudio/plugin-software-project` with role `core:delivery`. It fills three contribution points (`implementation-workflow`, `planning-methodology`, `review-process`) but should only fill `implementation-workflow`. Planning, review, and discovery are separate stages that belong in their own stage-definition plugins. The delivery workflow skeleton belongs in `agile-workflow` (the workflow-definition plugin), renamed from `delivery` to `agile-methodology`.

### V4: Connector may contain business logic

Connector hooks in `connectors/claude-code/src/hooks/` must be thin adapters: stdin → HookContext → daemon POST /hook → output. Any hook that evaluates enforcement rules, checks field values, or makes governance decisions locally instead of delegating to the daemon is a violation. The daemon is the business logic boundary (AD-1ef9f57c question 8).

### V5: Agent prompts not fully generated from pipeline

`.claude/agents/` files contain hardcoded content that should be pipeline output. Knowledge references list IDs without titles. No stage context (Layer 2) included. Pipeline should produce three-layer composition: universal role + stage context + domain knowledge (RES-d6e8ab11 section 4).

### V6: Required contribution points filled incorrectly

The delivery workflow has 3 required contribution points. Currently the software plugin fills all three, but planning and review should come from separate stage-definition plugins. Core learning-pipeline contribution is missing.

### V7: Rule enforcement not through daemon for all entry types

Rules don't declare entries for all daemon enforcement types (`field-check`, `tool-matcher`). Hand-written scripts duplicate daemon logic. Daemon is the business logic boundary (AD-1ef9f57c question 8).
