## What

Review existing team design research (RES-97e3aa4b) within the context of the product architecture decisions made in session 3. Keep what aligns, discard what doesn't, fill gaps. The new principles are documented in session state under "Product architecture (CRITICAL)".

## Research scope

### 1. End-to-end workflow definition
Define the complete project workflow (discovery → delivery → learning) and how it is COMPOSED from plugin installations:
- **core-framework plugin** — provides the workflow skeleton, artifact lifecycle, learning loop
- **discovery plugin** — fills in discovery stages (research, planning, documentation)
- **delivery plugin** — fills in delivery stages (implementation, review, release)
- The installed combination determines the project's specific workflow

### 2. Agent types per workflow stage
Each workflow stage has different TYPES of work. Agent types specialise in these:
- **Discovery stage examples (software):** technical researcher, product researcher, UX researcher, architect/planner
- **Delivery stage examples (software):** frontend implementer, backend implementer, mobile implementer, reviewer, designer
- **Learning stage:** governance steward, lesson analyst
- Agent types are provided by the three plugin layers (core, discovery, delivery), each owning their stages

### 3. Knowledge plugins slot into agent types
- Language/framework plugins (Rust, Svelte, etc.) register knowledge against agent types
- Domain plugins (legal, medical, etc.) do the same for non-software projects
- Knowledge is bidirectional: every knowledge artifact has agents, every agent has knowledge

### 4. Programmatic system prompt generation
- Agent team composition is determined by: workflow (from plugins) + project type + installed knowledge
- System prompts are GENERATED by composing agent type definition + applicable knowledge + project context
- When knowledge is added, affected system prompts regenerate
- Orchestrator's system prompt embeds the workflow — its sole job is to ensure the workflow is followed

### 5. State machine — redesign from scratch
- Current state machine predates the workflow architecture and doesn't fit (e.g., no "todo" for tasks, statuses don't map to workflow stages)
- Design a new state machine that emerges from the workflow definition
- Artifact statuses should reflect where work is in the workflow, not generic lifecycle labels
- Define canonical artifact type definitions: what IS a rule vs knowledge vs doc vs decision — clear criteria
- Migration from old → new state machine becomes an implementation task

### 6. Process requirements
- Every workflow step = one context window task (if it doesn't fit, planning wasn't thorough enough)
- Agents run quality checks (orqa check) but do NOT commit
- Orchestrator reviews and commits
