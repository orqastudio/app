# Plugin Schema Ingestion in LSP

## What to Implement

The LSP server currently calls `validate_file()` with an empty `artifact_types` slice. The daemon already holds `PluginContributions` in memory (loaded via `scan_plugin_manifests()` at startup and refreshed on `POST /reload`).

### Steps

1. **Add `GET /schema` endpoint to the daemon** (`libs/validation/src/daemon.rs`) — returns all `ArtifactTypeDef` entries from `DaemonState.plugin_contributions` as JSON.

2. **Add schema fetch to the LSP backend** (`libs/lsp-server/src/server.rs`) — on `initialize`, fetch `GET /schema` from the daemon and store the result in `OrqaLspBackend`.

3. **Re-fetch on plugin manifest save** — when `didSave` fires for a file matching `**/orqa-plugin.json`, call `POST /reload` then re-fetch `GET /schema`.

4. **Pass artifact types to `validate_file()`** — replace the empty slice with the fetched types.

## Acceptance Criteria

- [ ] Daemon exposes `GET /schema` returning all artifact type definitions as JSON
- [ ] LSP fetches schemas on `initialize` and stores them in backend state
- [ ] LSP re-fetches schemas when any `orqa-plugin.json` is saved
- [ ] `validate_file()` receives real artifact types from plugin schemas
- [ ] No `unwrap()` / `expect()` / `panic!()` in new code
- [ ] `make lint-backend` passes with zero warnings
- [ ] Existing tests continue to pass
