# Rule Enforcement Architecture

## 1. Executive Summary

OrqaStudio has 99 rule files in `.orqa/process/rules/`. These rules carry 138 enforcement entries across three current mechanism types: `behavioral` (60 entries across all 59 active rules), `hook` (44 entries across 20 rules), and several minor types (`pre-commit`, `lint`, `tool`, `svelte-check`, `cargo-test`, `vitest`, `tsc`). The `behavioral` mechanism is purely advisory -- injected as text in agent prompts. The `hook` entries in rule frontmatter describe *desired* hook enforcement but do not generate actual hooks. Instead, hooks are hand-written TypeScript in the connector.

The gap is clear: **20 rules already declare hook specifications in their frontmatter, but only 3 of those rules have corresponding hand-written hooks in the connector** (RULE-99abcea1 via `enforce-background-agents.mjs`, RULE-5d2d39b7 via `enforce-completion-gate.mjs`, and some bash safety rules from RULE-00700241/RULE-f609242f/RULE-0be7765e via `rule-engine.ts`). The remaining 17 rules with hook specifications are unenforced -- their frontmatter describes enforcement that does not exist.

This research proposes:

1. A **five-mode enforcement taxonomy** that formalizes what already exists informally
2. A **declarative constraint language** in YAML frontmatter that covers 80% of mechanical enforcement needs
3. A **hook generation pipeline** that runs at `orqa install` time and produces Claude Code hooks.json entries from rule specifications
4. A **template library** of parameterized hook patterns covering the six enforcement archetypes found in the existing rules

### Key Numbers

- 99 total rule files, 59 with `status: active`
- 138 total enforcement entries
- 44 hook entries declared in frontmatter, ~5-7 actually enforced by hand-written hooks
- 27 rules are pure-behavioral (only `mechanism: behavioral`, no mechanical enforcement)
- Estimated 18-22 rules are candidates for full mechanical enforcement
- Estimated 8-12 more are candidates for partial mechanical enforcement (some conditions mechanical, some behavioral)

## 2. Enforcement Mode Taxonomy

### Proposed Five Modes

| Mode | Description | Enforcement Time | Output Channel |
| ------ | ------------- | ----------------- | ---------------- |
| **mechanical** | Generates a hook that blocks or warns on tool use | Real-time (PreToolUse/PostToolUse) | Claude Code hook |
| **advisory** | Injected as compressed summary in agent prompt | Agent spawn time | Prompt pipeline |
| **validation** | Checked at commit time by linters/pre-commit hooks | Commit time | Pre-commit hook / `make check` |
| **workflow-guard** | Expressed as a guard on state machine transitions | Transition time | Workflow engine |
| **observational** | Logged for tracking, no enforcement action | Session lifecycle | Telemetry/metrics |

### How They Compose

A single rule can have entries across multiple modes. This is already the pattern -- RULE-0be7765e has `behavioral` + `pre-commit` + `hook` entries. The taxonomy formalizes the composition:

- **Safety rules** (RULE-00700241, RULE-f609242f): `mechanical` (PreToolUse blocks) + `advisory` (prompt text as backup)
- **Process rules** (RULE-99abcea1, RULE-5d2d39b7): `mechanical` (tool-use warnings) + `advisory` (prompt text)
- **Quality rules** (RULE-9814ec3c, RULE-42d17086): `validation` (linter config) + `advisory` (prompt text)
- **Artifact rules** (RULE-b10fe6d1, RULE-3c2da849): `workflow-guard` (state transitions) + `mechanical` (file protection) + `advisory`
- **Behavioral rules** (RULE-2f64cc63, RULE-43f1bebc): `advisory` only -- no mechanical enforcement possible

### Mode Decision Framework

```text
Can the rule be expressed as a condition on tool_input fields?
  YES --> mechanical (PreToolUse/PostToolUse)
  NO  --> Can the rule be checked by a linter or build tool?
            YES --> validation (pre-commit/make check)
            NO  --> Can the rule be expressed as a state transition guard?
                      YES --> workflow-guard (workflow engine)
                      NO  --> Is it about agent behavior/reasoning quality?
                                YES --> advisory (prompt injection)
                                NO  --> observational (logging only)
```

## 3. Rule-to-Hook Generation Design

### Recommendation: Parameterized Templates from Frontmatter Declarations

After reviewing the 44 existing hook entries and the 5 hand-written hooks, the enforcement patterns cluster into six archetypes. A **template-based approach** where rules parameterize hook templates via frontmatter is the recommended design. This avoids the complexity of a full constraint compilation language while covering the actual enforcement needs.

### Why Not a Constraint Language?

A constraint language (like a DSL that compiles to hooks) would be elegant but:

1. **PD-1ef9f57c resolved this**: "Declarative only. Field checks, relationship checks, graph queries. Code hooks for anything beyond that." The guard language is explicitly kept simple.
2. **The patterns are finite**: Six archetypes cover all 44 hook entries. A general language would be over-engineered.
3. **Debugging**: Template-generated hooks are readable TypeScript. Compiled hooks from a DSL are opaque.

### Why Not Pure Frontmatter Conditions?

A purely declarative system like the example in the research questions:

```yaml
enforcement:
  mode: mechanical
  event: PreToolUse
  matcher: Agent
  condition:
    field: tool_input.run_in_background
    operator: equals
    value: true
  action: block
```

This works for simple field checks but fails for:

- **Regex matching on bash commands** (the most common pattern -- 15+ entries)
- **File system checks** (completion gate reads `.state/session-state.md`)
- **Glob matching on file paths** (file protection, knowledge injection)
- **Conditional enforcement** (dogfood flag check)

The recommendation is a **hybrid**: frontmatter declares the enforcement template and its parameters, the generator selects and parameterizes the appropriate template.

## 4. The Six Enforcement Archetypes

Analysis of all 44 hook entries reveals six distinct patterns. Each becomes a template.

### Archetype 1: Bash Command Pattern Match

**Used by:** RULE-00700241 (7 entries), RULE-f609242f (8 entries), RULE-0be7765e (1 entry), RULE-d2c2063a (1 entry)
**Total entries:** 17 (39% of all hook entries)

```yaml
enforcement:
  - mode: mechanical
    template: bash-pattern
    event: PreToolUse
    matcher: Bash
    action: block|warn
    pattern: "regex pattern"
    message: "human-readable violation message"
```

**Generated hook logic:** Read `tool_input.command`, test against regex, emit block/warn.

**Current state:** Partially implemented in `rule-engine.ts` (4 hardcoded patterns). The remaining 13 declared patterns are NOT enforced.

### Archetype 2: Tool Input Field Check

**Used by:** RULE-99abcea1 (Agent must have `run_in_background: true` and `team_name`)
**Total entries:** 2 (but could apply to many more rules)

```yaml
enforcement:
  - mode: mechanical
    template: field-check
    event: PreToolUse
    matcher: Agent
    checks:
      - field: tool_input.run_in_background
        operator: equals
        value: true
        message: "Agents must run in background"
      - field: tool_input.team_name
        operator: exists
        message: "Agents must specify team_name"
    action: warn
```

**Generated hook logic:** Read `tool_input`, check each field condition, emit warn/block.

**Current state:** Implemented in `enforce-background-agents.mjs` (hand-written).

### Archetype 3: File Path Guard

**Used by:** RULE-3c2da849 (2 entries), RULE-8ee65d73 (2 entries), RULE-af5771e3 (1 entry), RULE-998da8ea (1 entry)
**Total entries:** 6 (14% of all hook entries)

```yaml
enforcement:
  - mode: mechanical
    template: file-guard
    event: PreToolUse|PostToolUse
    matcher: Write|Edit
    file_patterns:
      - ".orqa/implementation/*/schema.json"
      - ".orqa/process/*/schema.json"
    action: warn|block
    message: "Core graph artifacts are firmware"
    condition:
      dogfood: false  # optional: skip when dogfood mode is active
```

**Generated hook logic:** Read `tool_input.file_path`, glob-match against patterns, check optional conditions (like dogfood flag), emit warn/block.

**Current state:** Partially implemented in `rule-engine.ts` (plugin ownership and secrets only). The rule-declared file patterns are NOT enforced.

### Archetype 4: Session State Check

**Used by:** RULE-5d2d39b7 (completion gate before TeamCreate)
**Total entries:** 1, but the pattern is reusable

```yaml
enforcement:
  - mode: mechanical
    template: session-state-check
    event: PreToolUse
    matcher: TeamCreate
    check: completion-gate  # named check from a fixed vocabulary
    action: warn
    message: "Resolve outstanding items before creating a new team"
```

**Generated hook logic:** Read `.state/session-state.md`, run the named check (e.g., look for unchecked items in `### Completion Gate` section), emit warn if issues found.

**Current state:** Implemented in `enforce-completion-gate.mjs` (hand-written).

### Archetype 5: Knowledge Injection

**Used by:** RULE-e1f1afc1 (7 entries), RULE-43f1bebc (1 entry), RULE-71352dc8 (1 entry), RULE-49f66888 (1 entry), RULE-998da8ea (1 entry)
**Total entries:** 11 (25% of all hook entries)

```yaml
enforcement:
  - mode: mechanical
    template: knowledge-inject
    event: PostToolUse
    matcher: Write|Edit
    file_patterns:
      - "backend/src-tauri/src/domain/**"
    knowledge:
      - orqa-domain-services
      - orqa-error-composition
```

**Generated hook logic:** Check if the tool output file path matches any pattern, if so inject the named knowledge artifacts as a systemMessage.

**Current state:** Implemented in `knowledge-injector.ts` via the prompt pipeline -- but the file-path-to-knowledge mappings are in the prompt registry, not dynamically generated from rule frontmatter. The rules declare intent; the pipeline has its own separate configuration.

### Archetype 6: Session Lifecycle

**Used by:** RULE-87ba1b81 (Stop check), RULE-5dd9decd (Stop check), RULE-145332dc (Stop check), RULE-30a223ca (Stop + SessionStart), RULE-c603e90e (SessionStart surface), RULE-998da8ea (SessionStart inject)
**Total entries:** 7 (16% of all hook entries)

```yaml
enforcement:
  - mode: mechanical
    template: session-lifecycle
    event: Stop|SessionStart
    action: check|surface|inject
    description: "Stop hook verifies X"
```

**Generated hook logic:** These are more varied and harder to template. The Stop hook is currently a monolithic bash script that performs multiple checks. SessionStart similarly. Generated hooks here would need to contribute *sections* to the existing lifecycle scripts rather than replacing them.

**Current state:** Implemented in `stop-checklist.sh` and `session-start.sh` (hand-written monoliths).

### Archetype Distribution

| Archetype | Entry Count | Percentage | Template Feasibility |
| ----------- | ------------- | ------------ | --------------------- |
| Bash Command Pattern | 17 | 39% | HIGH -- pure regex, trivially templateable |
| Tool Input Field Check | 2 | 5% | HIGH -- field/operator/value, trivially templateable |
| File Path Guard | 6 | 14% | HIGH -- glob match + optional condition |
| Session State Check | 1 | 2% | MEDIUM -- limited vocabulary of checks |
| Knowledge Injection | 11 | 25% | MEDIUM -- needs integration with prompt registry |
| Session Lifecycle | 7 | 16% | LOW -- monolithic scripts, contribution model needed |

**Summary:** 58% of hook entries (archetypes 1-3) can be fully generated from templates with high confidence. Another 27% (archetypes 4-5) can be generated with medium confidence. The remaining 16% (archetype 6) requires a different approach (contributing sections to lifecycle scripts).

## 6. Hook Generation Pipeline

### When: Install Time

Generation happens during `orqa install` (specifically during `orqa plugin install` for the claude-code connector). This aligns with the research principle that resolved workflows are files on disk, not runtime computations.

### Pipeline Steps

```text
1. SCAN: Read all active rules from .orqa/process/rules/ and plugin rules/ directories
         Filter to rules with enforcement entries where mode = "mechanical" (or mechanism = "hook")

2. PARSE: For each rule, extract enforcement entries
          Validate against JSON schema for the enforcement entry format
          Map each entry to its archetype (bash-pattern, field-check, file-guard, etc.)

3. GENERATE: For each entry, select the template for its archetype
             Parameterize the template with the entry's fields (pattern, action, message, etc.)
             Produce a hook entry (event + matcher + command path)

4. MERGE: Combine generated hook entries with hand-written hook entries
          Hand-written hooks ALWAYS take precedence (override by matcher key)
          Deduplication by (event, matcher, rule-id)

5. WRITE: Write the merged hooks to the connector's hooks.json
          Write generated hook scripts to connector's dist/hooks/generated/
          Write a manifest of what was generated for debugging

6. VERIFY: Run each generated hook with a no-op input to verify it loads without errors
           Report generation stats (N rules processed, M hooks generated, K conflicts)
```

### Where Generated Hooks Live

```text
connectors/claude-code/
  hooks/
    hooks.json                           # Merged: hand-written + generated
    scripts/
      enforce-background-agents.mjs      # Hand-written (kept as-is)
      enforce-completion-gate.mjs        # Hand-written (kept as-is)
      session-start.sh                   # Hand-written (kept as-is)
      stop-checklist.sh                  # Hand-written (kept as-is)
      stability-check.mjs               # Hand-written (kept as-is)
    generated/
      RULE-f609242f-git-no-verify.mjs    # Generated from rule
      RULE-f609242f-git-force-push.mjs   # Generated from rule
      RULE-00700241-bash-sudo.mjs        # Generated from rule
      RULE-3c2da849-core-protection.mjs  # Generated from rule
      ...
      _manifest.json                     # What was generated and from which rules
```

### How Generated Hooks Coexist with Hand-Written Hooks

The merge step uses a precedence system:

1. **Hand-written hooks** in `hooks/scripts/` are never overwritten or removed
2. **Generated hooks** go in `hooks/generated/` (separate directory)
3. The `hooks.json` is rewritten on each generation, combining both sources
4. If a hand-written hook and a generated hook would target the same (event, matcher) pair, the hand-written hook wins and a warning is logged
5. Generated hooks include a comment header: `// Generated from RULE-{id} -- do not edit manually`

### Claude Code Hook Format Constraints

Claude Code hooks follow this format (from `.claude/settings.json` or `hooks.json` in a plugin):

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node path/to/script.mjs",
            "timeout": 5000
          }
        ]
      }
    ]
  }
}
```

Key constraints:

- **matcher** is a string (tool name or pipe-separated list like `Write|Edit|Bash`)
- **command** must be a shell command (node, bash, etc.)
- **timeout** is in milliseconds
- The script receives JSON on stdin (`{ tool_name, tool_input }` for PreToolUse)
- The script writes JSON to stdout: `{}` (allow), `{ "decision": "block", "reason": "..." }` (block), or `{ "systemMessage": "..." }` (warn/inject)

These constraints mean:

- Each hook entry is one script invocation
- Multiple rules targeting the same (event, matcher) should be combined into a single script for efficiency
- Generated scripts must handle stdin/stdout in the Claude Code format

### Optimization: Grouped Rule Scripts

Instead of one script per rule entry (which could mean 17 scripts for bash patterns alone), the generator should group entries by (event, matcher) and produce one script per group:

```text
generated/
  pretooluse-bash-rules.mjs        # All bash-pattern rules (17 entries)
  pretooluse-agent-rules.mjs       # All Agent field-check rules
  pretooluse-write-edit-rules.mjs  # All file-guard rules
  posttooluse-write-edit-rules.mjs # All PostToolUse file checks
```

Each grouped script contains all the rules for its (event, matcher) pair, checks them all in sequence, and returns the highest-severity result (block > warn > allow).

## 7. Declarative Constraint Language

### Frontmatter Schema

The proposed enforcement entry schema extends what already exists in rule frontmatter. The existing format is informal -- different rules use different field names. The proposal standardizes it.

```yaml
enforcement:
  - mode: mechanical|advisory|validation|workflow-guard|observational
    # For mechanical mode:
    template: bash-pattern|field-check|file-guard|session-state-check|knowledge-inject|session-lifecycle
    event: PreToolUse|PostToolUse|UserPromptSubmit|SessionStart|Stop|PreCompact|SubagentStop
    matcher: "Bash|Agent|Write|Edit|TeamCreate|TaskUpdate|*"
    action: block|warn|inject|check|surface
    # Template-specific parameters:
    pattern: "regex"              # bash-pattern
    message: "human message"      # all templates
    checks: [...]                 # field-check
    file_patterns: [...]          # file-guard
    knowledge: [...]              # knowledge-inject
    check: "named-check"          # session-state-check
    condition:                    # optional conditional
      dogfood: true|false
      role: "orchestrator"
```

### JSON Schema for Validation

A JSON Schema should validate enforcement entries at `orqa install` time. Invalid entries produce warnings, not failures (to avoid blocking install on schema evolution).

### Backward Compatibility with Existing Frontmatter

The existing 44 hook entries use fields like `type`, `event`, `action`, `pattern`, `description`, `knowledge`, and `condition`. The new schema maps cleanly:

| Existing Field | New Field | Notes |
| ---------------- | ----------- | ------- |
| `mechanism: hook` | `mode: mechanical` | Rename |
| `type: PreToolUse` | `event: PreToolUse` | Rename |
| `event: bash` | `matcher: Bash` | The existing "event" was really the matcher |
| `action: block` | `action: block` | Same |
| `pattern: "..."` | `pattern: "..."` | Same |
| `description: "..."` | (dropped) | Use `message` instead |
| `knowledge: "x, y"` | `knowledge: [x, y]` | String to array |
| `condition: "dogfood: true"` | `condition: { dogfood: true }` | String to object |

Migration of existing frontmatter is a one-time task during the generation pipeline implementation.

### Limits of the Declarative Language

The declarative language CANNOT express:

1. **Multi-step checks** -- e.g., "if this is a git commit AND the commit includes .rs files, warn about binary rebuild" (RULE-aff3c5db). This requires reading staged files, not just the command string.
2. **Stateful checks** -- e.g., "if a team exists from a previous batch, warn before creating a new one" requires tracking team state across tool calls.
3. **Content analysis** -- e.g., "check if a file contains TODO patterns" (RULE-af5771e3 PostToolUse check). This requires reading the file content after write.
4. **Complex boolean logic** -- e.g., "block if (dogfood AND file in plugins/) OR (not dogfood AND file in .orqa/process/agents/)".

For these cases, the rule should declare `mode: mechanical` with `template: custom` and a `script` field pointing to a hand-written hook script. The generator treats custom-template entries as pass-through (it adds them to hooks.json but does not generate the script).

## 8. Integration with Existing Systems

### Workflow Engine Guards

Workflow guards and mechanical hooks serve different enforcement layers:

| Concern | Enforcer | When |
| --------- | ---------- | ------ |
| "Can this artifact transition from state A to state B?" | Workflow engine guard | At transition request |
| "Is this tool call safe/permitted?" | Mechanical hook | At tool invocation |
| "Does this code meet quality standards?" | Validation (linter) | At commit time |
| "Does the agent know this rule?" | Advisory (prompt) | At agent spawn |

There is no overlap to resolve. Workflow guards check artifact state; hooks check tool use. A rule like RULE-b10fe6d1 (artifact lifecycle) uses workflow guards for state transitions AND prompt advisory for behavioral guidance. The two modes compose naturally.

### Prompt Pipeline

The prompt pipeline currently handles advisory enforcement. Rules with `mode: advisory` continue to be processed by the prompt pipeline. Rules with `mode: mechanical` are processed by the hook generator. Rules with both modes get both treatments -- the prompt reminds the agent, and the hook catches violations.

The prompt pipeline should be modified to **include a note about mechanical enforcement** in the agent prompt when a rule has both modes:

```text
RULE-f609242f (Git Workflow): Mechanically enforced via PreToolUse hook.
  Block: --no-verify, force-push to main, reset --hard, clean -f.
  Warn: --force-with-lease, branch -D, checkout ./restore .
```

This tells the agent that the rule has teeth, which empirically improves compliance even before the hook fires.

### Validation Crate

The Rust validation crate handles commit-time checks (structural validation, relationship integrity, schema conformance). Rules with `mode: validation` map to validators in the crate. This is a separate concern from hook generation and does not need changes for this proposal.

### Connector Hooks

The existing connector hooks (`rule-engine.ts`, `knowledge-injector.ts`, etc.) should be **refactored to consume generated rule data** rather than hardcoding rules:

1. `rule-engine.ts` currently hardcodes 4 bash safety patterns. After generation, it should read from a generated data file or be replaced entirely by the generated grouped script.
2. `knowledge-injector.ts` currently reads from the prompt registry. After generation, the file-path-to-knowledge mappings from RULE-e1f1afc1 should flow through the same pipeline.

### Generation Flow Diagram

```text
.orqa/process/rules/RULE-*.md        connectors/claude-code/hooks/scripts/*
          |                                        |
          v                                        v
    [Rule Scanner]                        [Hand-written hooks]
          |                                        |
          v                                        |
  [Template Generator]                             |
          |                                        |
          v                                        v
  generated/*.mjs  ------>  [Merger]  <------- scripts/*.mjs
                               |
                               v
                         hooks/hooks.json
```

```yaml
enforcement:
  - mechanism: hook
    type: PreToolUse
    event: bash
    action: block
    pattern: "git\\s+(commit|push)\\b[^|&;]*--no-verify"
  - mechanism: hook
    type: PreToolUse
    event: bash
    action: block
    pattern: "git\\s+push\\b[^|&;]*(-f\\b|--force\\b)[^|&;]*(main|master)\\b"
```

**Migrated frontmatter:**

```yaml
enforcement:
  - mode: advisory
    message: "Use worktree-based workflow with mandatory cleanup"
  - mode: mechanical
    template: bash-pattern
    event: PreToolUse
    matcher: Bash
    action: block
    pattern: "git\\s+(commit|push)\\b[^|&;]*--no-verify"
    message: "Bypassing git hooks with --no-verify is forbidden (RULE-f609242f)"
  - mode: mechanical
    template: bash-pattern
    event: PreToolUse
```

## 10. Open Questions

### Q1: Agent Role in Hook Input

Claude Code hook input includes `tool_name` and `tool_input`, but it is unclear whether the agent's role (orchestrator, implementer, reviewer, etc.) is available. If not, role-based enforcement (RULE-b723ea53) cannot be mechanized. **Resolution needed: check Claude Code SDK documentation for hook input schema.**

### Q2: Session Lifecycle Hook Architecture

The current Stop and SessionStart hooks are monolithic scripts that perform many checks. Should the generation pipeline:

- (a) Add entries to the monolithic scripts (fragile, merge conflicts)
- (b) Generate separate scripts and add them as additional hook entries (simpler, more entries)
- (c) Refactor the monolith into a runner that loads check modules (cleanest, most work)

**Recommendation:** Option (b) for now, option (c) as a follow-up. Generated session lifecycle hooks can be separate entries in hooks.json; they do not need to be part of the monolithic scripts.

### Q3: Performance Impact

With 17+ bash pattern rules, each Bash tool call would test against all patterns. Current hook timeout is 5000ms. Is this a concern?

**Analysis:** Regex matching is sub-millisecond per pattern. 20 patterns tested sequentially takes <1ms. The stdin/stdout overhead dominates at ~10-50ms for Node.js process startup. Grouping all bash patterns into one script (one process spawn per Bash tool call) keeps the total under 100ms. **Not a concern.**

### Q4: Rule Frontmatter Migration

99 rule files need their enforcement entries migrated to the new schema. Should this be:

- (a) A one-time migration task
- (b) Gradual migration (old format supported alongside new)
- (c) New format only, old rules still work via backward-compatible parser

**Recommendation:** Option (a). The existing format is not formally validated, so there is no compatibility contract. Migrate all 59 active rules in one batch, validate with the new JSON schema, and commit.

## 11. Recommendations

### Immediate Actions

1. **Define the enforcement entry JSON schema** -- formalize the frontmatter format described in section 7.
2. **Migrate existing rule frontmatter** -- update all 59 active rules to the new schema. This is a mechanical transformation with well-defined mappings (section 7).
3. **Build the bash-pattern template** -- this alone covers 39% of all hook entries and is the simplest template to implement. A single generated script replaces hardcoded patterns in `rule-engine.ts`.
4. **Build the field-check template** -- covers the Agent tool checks. Replaces `enforce-background-agents.mjs`.
5. **Build the file-guard template** -- covers core file protection. Extends `rule-engine.ts`.

### Medium-Term Actions

1. **Build the generation pipeline** -- the scanner, template engine, merger, and writer described in section 6.
2. **Integrate with `orqa install`** -- trigger generation during connector plugin installation.
3. **Update the prompt pipeline** -- add mechanical enforcement notes to advisory prompts.
4. **Build the knowledge-inject template** -- unifies RULE-e1f1afc1's mappings with the knowledge injector.

### Long-Term Actions

1. **Refactor session lifecycle hooks** -- decompose monolithic scripts into modular check runners.
2. **Add enforcement coverage reporting** -- `orqa audit enforcement` shows which rules have mechanical enforcement vs. advisory only.
3. **Track enforcement effectiveness** -- log hook fires per rule, measure compliance rates, feed into the learning loop.

### What NOT To Do

- Do not build a constraint DSL or compilation language. The template approach is sufficient and debuggable.
- Do not try to mechanize judgment-based rules (vision alignment, systems thinking, continuous operation). These are advisory by nature.
- Do not replace hand-written hooks that work. The generation pipeline augments; it does not replace working code.
- Do not block `orqa install` on enforcement schema validation errors. Warn and skip invalid entries.
