# Task 7: Review of Existing Research Against New Design Principles

## Summary

RES-12f2bf80 (Persistent Dev Team Model) and RES-138eff6e (Token Efficiency Analysis) were evaluated against the seven new design principles from TASK-61605174. The existing research provides a solid foundation for composability and knowledge architecture but fundamentally conflicts with the new principles on agent persistence, prompt generation, workflow architecture, and state machines. The token efficiency research correctly diagnosed the problems but proposed patch-level fixes where the new principles demand architectural solutions.

## Detailed Analysis: RES-12f2bf80

### Executive Summary — Partially Aligns

**What aligns:** The four failure modes are accurately identified and still real problems:

1. Zero context continuity — still a problem, but the new solution is better prompts per task, not persistent agents
2. Resource contention — still a problem, solved the same way (sequential compilation)
3. Poor shutdown discipline — eliminated by one-context-window-per-task (no lingering agents)
4. Sequential blocking — addressed by workflow-derived handoffs, not persistent team communication

**What conflicts:** The proposed solution (persistent specialists) directly contradicts "one context window per task." The new principle says: if a task doesn't fit in one context window, the planning wasn't thorough enough. Persistent agents are the wrong abstraction — better task decomposition is the right one.

### Part 0: Plugin-Based Architecture — Aligns

This is the strongest alignment. The new principles explicitly preserve and extend this:

- **0.1 Composability Model** — Directly maps to new principle: "workflow COMPOSED from plugin installations." The insight that team structure comes from plugins, not framework, is preserved. The difference is that the new design composes the *workflow* from plugins, and agent types emerge from workflow stages rather than being pre-defined in a software plugin.

- **0.2 Knowledge Plugin Architecture** — Maps directly to new principle 3: "Knowledge plugins as composable expertise." Language plugins providing domain expertise that slots into agent types is exactly the new design. The bidirectional registration (agents have knowledge, knowledge has agents) is a new principle addition that strengthens this.

- **0.3 Plugin Load Order** — The three-layer model (framework → language plugins → workflow plugins) is preserved. The naming changes (software plugin → delivery plugin, adding discovery plugin) but the compositional principle is identical.

### Part 1: Current State Analysis — Aligns

All analysis is factually accurate and remains valid:

- Agent structure analysis (1.1) correctly maps the current landscape
- Knowledge architecture (1.2) correctly describes what exists
- Codebase technical domains (1.3) are accurate
- Failure modes (1.4) are all still real

No changes needed. This is diagnostic, not prescriptive.

### Part 2: Recommended Team Structure — Conflicts

**What conflicts:**

The core proposal of "persistent role-based team of 4-6 specialists" fundamentally conflicts with:

1. **One context window per task** — Persistent agents accumulate context across tasks. The new principle says each task gets a fresh, precisely-scoped context window. No accumulation. No drift. No context pollution from previous tasks.

2. **Workflow-stage-derived roles** — The research proposes fixed roles (Backend Lead, Frontend Lead, Integration Lead, Governance Lead). The new principles say agent types emerge from workflow stages: discovery stage produces researchers/planners, delivery stage produces implementers/reviewers. The "Lead" abstraction doesn't exist — there are agent *types* per workflow stage, and the specific agents spawned depend on what the workflow step requires.

3. **Programmatic prompt generation** — Persistent agents would need their prompts updated mid-session as context changes. Generated prompts per task are simpler, more precise, and waste fewer tokens because they contain exactly what's needed for that one task.

**What partially aligns:**

The *ownership boundaries* are sound: backend code owned by backend-specialized agents, frontend by frontend-specialized agents. This maps to the new principle of agent types per workflow stage — a "backend implementer" agent type gets backend knowledge. The difference is that it's spawned fresh per task, not persistent.

The communication model (2.3) with sequential handoffs through Integration Lead maps to workflow steps. But instead of agents talking to each other via SendMessage, the orchestrator embeds the handoff protocol in the workflow itself.

### Part 3: Expertise Mapping — Partially Aligns

**What aligns:**

The knowledge loading patterns per agent type are excellent and directly carry forward:

- Backend Lead's knowledge list (3.1) → becomes the knowledge loaded into a "backend implementer" agent type
- Frontend Lead's knowledge list (3.2) → becomes knowledge for "frontend implementer" agent type
- Integration Lead's knowledge list (3.3) → becomes knowledge for a cross-boundary verification step
- Governance Lead's knowledge list (3.4) → becomes knowledge for "governance steward" agent type

The ownership boundaries and domain separation are all preserved.

**What conflicts:**

- Fixed "Lead" role identities don't exist in the new model. There's no "Backend Lead" — there's an "implementer" agent type that, for a Rust backend task, gets Rust + Tauri + backend domain knowledge composed into its prompt.
- The Orchestrator section (3.5) loads "all active rules" at session start — directly contradicts token efficiency by design. The orchestrator gets the workflow embedded in its system prompt, not 58 rule files.

### Part 4: Workflow Model — Partially Aligns

**What aligns:**

The sequential feature implementation workflow (4.1) is sound engineering:

```text
Planner → Backend → Integration → Frontend → Integration → Reviewer → Governance
```

This maps to workflow stages. Each arrow is a workflow step with a defined agent type.

Parallel safety zones (4.2) and resource safety (4.3) are directly preserved.

**What conflicts:**

The workflow is hardcoded in the research document. The new principle says workflows are COMPOSED from plugin installations:

- core-framework plugin provides the workflow skeleton
- discovery plugin fills in discovery stages (research, planning, docs)
- delivery plugin fills in delivery stages (implementation, review, release)

The specific workflow from 4.1 would be what the software-delivery plugin defines. A non-software project (writing, design, legal) would have completely different workflow stages.

### Part 5: Knowledge Requirements — Partially Aligns

**What aligns:**

Knowledge loading at task start (5.1) is preserved. Each agent loads its knowledge before work begins. The specific knowledge lists are accurate and carry forward.

**What conflicts:**

Knowledge deduplication (5.2) assumes persistent agents ("Backend Lead carries knowledge across 3 tasks without re-learning"). In the new model, each task spawns a fresh agent with precisely the knowledge it needs. There's no dedup needed because there's no re-injection — each agent starts fresh with exactly the right knowledge composed into its generated prompt.

This is actually a token efficiency WIN: instead of loading 7 knowledge artifacts for a persistent agent (some irrelevant to the current task), the generated prompt includes only the 2-3 artifacts relevant to THIS specific task.

### Part 6: Team Size and Composition — Conflicts

The entire section assumes a persistent team of 5 agents. The new model doesn't have a fixed team size — it has a workflow with steps, and each step spawns the agent type it needs. A simple bug fix might need one implementer agent. A complex feature implementation might need 4-5 sequential agents across workflow stages. The "team" is the workflow execution, not a standing group.

### Part 7: Agent Lifecycle — Conflicts

Session-long persistence is the direct opposite of one-context-window-per-task. The new lifecycle is:

1. Orchestrator reads workflow step
2. Orchestrator generates system prompt for that step (agent type + knowledge + context)
3. Agent spawns, works, produces output
4. Agent terminates (context window done)
5. Orchestrator reads output, advances workflow to next step

This eliminates: stale agents, context drift, shutdown discipline problems, resource contention from idle agents.

### Part 8: Anti-Patterns — Partially Aligns

**Still valid anti-patterns:**

- Parallel Rust compilation (8.2) — still dangerous, same fix
- No handoff discipline (8.1, #4) — solved by workflow steps instead of SendMessage
- Agents using cargo check instead of orqa check (8.1, #7) — still valid, enforced via generated prompts
- Agents not knowing quality workflow (8.1, #6) — solved by generated prompts that embed the quality workflow

**Superseded anti-patterns:**

- "Spawn agents per task without team concept" (8.1, #1) — the new model IS task-scoped agents, but with generated prompts that include all needed knowledge. The knowledge re-loading problem is solved by prompt generation, not persistence.
- "Stale agents consuming resources" (8.1, #3) — eliminated by one-context-window-per-task design.

### Part 9: Implementation Plan — Conflicts

Nearly all implementation steps assume persistent agents and the software plugin team structure. The new implementation would:

- Create workflow definitions in discovery/delivery plugins instead of agent definitions in software plugin
- Create a prompt generator instead of static agent YAML files
- Define state machines per artifact type in the plugin that owns them
- Build orchestrator workflow enforcement instead of team management

### Part 9.6: Stack Adaptation — Aligns

The composability principle is directly preserved. The mechanism changes (swap knowledge in generated prompts vs. swap knowledge references in agent YAML) but the principle is identical: same workflow + different knowledge = different project type.

### Part 11: Success Metrics — Partially Aligns

The goals are right, the measurement mechanism needs updating:

- "Context continuity" → becomes "prompt quality" (does each agent get what it needs?)
- "Resource contention drops" → same metric, same solution
- "Agent shutdown cleans up" → becomes "no stale agents" (trivially solved by task-scoped lifetime)
- "Handoff discipline" → becomes "workflow step handoffs work"
- "Session quality" → same
- "Learning accumulates" → same (via lessons, not agent memory)

## Detailed Analysis: RES-138eff6e

### Diagnosis — All Valid

The token efficiency research's diagnostic sections (1-8) are excellent and all findings remain valid as a baseline:

- **48K tokens for 58 rules** — this is the cost of the current approach and the benchmark to beat
- **9,500-16,500 tokens per turn** — this is the overhead the new architecture must eliminate
- **63K tokens per 8-agent team spawn** — this shows the scale of duplication
- **13.4x overhead** (1,600 theoretical vs 21,400 actual) — this is the gap the new design must close
- **~60K tokens/session wasted** — this is the saving opportunity

These numbers should be preserved as the "before" measurement.

### Recommendation 1: Lazy Rule Loading — Superseded

The recommendation proposes using semantic search to find 3-4 relevant rules per turn instead of loading all critical rules.

**Why superseded:** The new architecture eliminates rule files from agent prompts entirely. Instead of loading rules as separate files and selecting which to inject, the workflow + agent type + knowledge compose into a generated prompt that already contains exactly the right constraints. There are no "58 rule files" in the new architecture — there are workflow definitions, agent type definitions, and knowledge artifacts that compose into prompts.

The savings estimate (35K tokens/session) is still valid but achieved through architecture, not optimization.

### Recommendation 2: Orchestrator Context Discipline — Still Relevant

The recommendation to keep agent findings on disk and not read them into orchestrator context is directly applicable regardless of architecture. The orchestrator in the new model still coordinates and still risks accumulating implementation details.

This should carry forward as a design constraint: orchestrator reads task output summaries (structured, brief), not full findings.

### Recommendation 3: Agent Model Tiering — Still Relevant

Using cheaper/faster models for simpler tasks is orthogonal to the architecture redesign and still valuable. The new architecture makes this easier: since prompts are generated per task, the prompt generator can also select the model tier based on task complexity.

Model tiering should be a first-class feature of the prompt generator.

### Recommendation 4: Selective Knowledge Injection — Partially Superseded

Knowledge deduplication and relevance filtering are still needed, but the mechanism changes. Instead of a runtime knowledge injector hook, the prompt generator assembles knowledge at generation time. The MIN_SCORE threshold and dedup logic should be implemented in the generator, not in a hook.

### Recommendation 5: Agent Shutdown Enforcement — Superseded

One-context-window-per-task eliminates stale agents entirely. Each agent runs, completes, and its context window closes. There is no agent to be stale. The 1-2K token saving is achieved by architecture, not enforcement.

## Gaps: What Neither Research Document Covers

### Gap 1: Workflow Composition from Plugins

Neither document describes how the end-to-end workflow (discovery -> delivery -> learning) is composed from plugin installations. RES-12f2bf80 describes a fixed team structure; it doesn't describe a composable workflow. The new design requires:

- core-framework plugin provides workflow skeleton
- discovery plugin fills discovery stages
- delivery plugin fills delivery stages
- The installed combination determines the workflow

This is entirely new research.

### Gap 2: Programmatic System Prompt Generation

Neither document describes how system prompts are generated from agent type + knowledge + project context + workflow position. RES-12f2bf80 uses static YAML frontmatter for agent definitions. The new design requires:

- A prompt generator that composes agent type definitions + applicable knowledge + project context
- When knowledge is added, affected prompts regenerate
- Orchestrator's prompt embeds the workflow

This is the core technical mechanism of the new design and has no coverage.

### Gap 3: State Machine Redesign

Neither document addresses state machines. The new principles require:

- State machine definitions owned by the plugin that supplies the artifact type
- Deterministic transitions (one valid next status per action)
- Human gates at review points with defined sub-workflows
- Ad-hoc workflows for bugs, UX tweaks, small fixes
- Statuses that reflect workflow position, not generic lifecycle labels

This is entirely new research.

### Gap 4: Discovery Plugin Design

RES-12f2bf80 focuses exclusively on the delivery side (implementation, review). The new principles introduce a discovery plugin that fills in discovery stages:

- Technical research, product research, UX research
- Architect/planner agents
- Documentation before implementation

The discovery side of the workflow is not covered by either document.

### Gap 5: Token Usage Observability

RES-138eff6e measures current waste but doesn't propose observability tooling for the new architecture. The new design needs:

- Token usage tracking per session/task/agent
- Identification of waste patterns
- Metrics for improvement over time
- Dashboard or reporting mechanism

### Gap 6: Ad-Hoc Workflows

Neither document covers fast-track workflows for unplanned work (bug reports, UX tweaks, small fixes). The new principles explicitly require these: "Not everything goes through full discovery -> delivery. Bug reports, UX tweaks, and small fixes need fast-track workflows that still have appropriate quality gates but skip heavy planning."

### Gap 7: Review Sub-Workflows

Neither document defines the internal structure of human gates. The new principles require:

- What gets reviewed (acceptance criteria, diff, findings)
- Who reviews (human, agent, both — and in what order)
- What the review produces (verdict, feedback, lessons)
- What happens on FAIL vs PASS
- Sub-workflows defined by the delivery plugin

3. **Model tiering** (Rec 3) — Cheaper models for simpler tasks. The prompt generator should also select model tier.

4. **Knowledge relevance filtering** (Rec 4) — MIN_SCORE thresholds, deduplication logic. These apply to the prompt generator's knowledge selection.

## Discard List

These specific decisions, patterns, and assumptions from the existing research conflict with the new principles and should NOT be carried forward:

### From RES-12f2bf80

1. **Persistent agent team** — The core proposal. Agents persisting for entire sessions conflicts with one-context-window-per-task. Discard in favor of task-scoped agents with generated prompts.

2. **Fixed team of 5 agents** (Part 6) — Team size should be determined by workflow, not predetermined. A bug fix needs 1-2 agents. A feature implementation needs 4-6 sequential agents. No fixed number.

3. **SendMessage-based handoffs** (Part 2.3) — Inter-agent communication via messaging assumes concurrent persistent agents. In the new model, the orchestrator manages handoffs by reading previous agent output and composing the next agent's prompt. No inter-agent messaging needed.

4. **Session-long agent lifecycle** (Part 7) — Agents spawn for a task, work, and terminate. No session-long persistence. No "brief team on scope at session start."

5. **Knowledge deduplication via session cache** (Part 5.2) — No session-spanning agents means no dedup cache. Each agent gets exactly the knowledge it needs in its generated prompt. Fresh every time.

6. **"Interview domain experts" pattern** (Part 4.1) — Planner interviewing Backend/Frontend leads assumes persistent agents with accumulated knowledge. In the new model, the planner agent gets the relevant knowledge in its prompt.

7. **Specialist pool as separate from core team** (Part 2.1) — The distinction between "core team" (persistent) and "specialist pool" (spawned as needed) is eliminated. All agents are task-scoped. The workflow determines what agent types are needed at each step.

### From RES-138eff6e

1. **Lazy rule loading** (Rec 1) — Superseded by prompt generation. Rules don't exist as separate files loaded into context; they're embedded in workflow definitions.

2. **Agent shutdown enforcement** (Rec 5) — Superseded by task-scoped agents. No stale agents to shut down.

3. **Prompt injector hook optimization** (Section 4) — The prompt injector hook becomes unnecessary. Prompt generation replaces runtime injection.

4. **Patching the current rule loading mechanism** — All recommendations about improving the 58-rule loading pattern are superseded. The pattern itself is eliminated.

## Migration Considerations

### From Current Architecture to New Design

1. **Rules -> Workflow Definitions**: 58 rule files need to be analyzed for which constraints belong in:
   - Workflow definitions (process rules like delegation, documentation-first)
   - Agent type definitions (behavioral rules like honest-reporting, no-stubs)
   - Knowledge artifacts (domain rules like coding-standards, error-handling)
   - Eliminated (enforcement rules that become unnecessary with generated prompts)

2. **Agent Definitions -> Agent Types**: Current agent YAML files in `.orqa/process/agents/` become agent type templates in plugins. The Backend Lead agent definition becomes the "backend implementer" agent type in the delivery plugin. Knowledge references become the prompt generator's knowledge selection rules.

3. **Prompt Injector -> Prompt Generator**: The current `prompt-injector.ts` hook (605 lines of runtime injection) is replaced by a build-time prompt generator that composes agent type + knowledge + context into a complete prompt. This is a fundamental architecture change, not a refactor.

4. **State Machines**: Current status transitions (scattered across rules and the artifact lifecycle rule) must be extracted into plugin-owned state machine definitions. Each plugin that provides an artifact type defines that type's state machine.

5. **Orchestrator Prompt**: The current 370-line CLAUDE.md + 58 injected rules becomes a generated prompt that embeds the workflow. The orchestrator's job simplifies to: follow the workflow, spawn the right agent type at each step, review output, commit.

6. **Team Management -> Workflow Execution**: TeamCreate/TeamDelete lifecycle becomes workflow step execution. Instead of "create a team of 5 agents," it's "execute this workflow step by spawning the appropriate agent type."

7. **Knowledge Injection -> Prompt Composition**: The current PostToolUse knowledge injector hook becomes part of the prompt generator. Knowledge is selected at prompt generation time, not injected at runtime.

### Risk Areas

- **Rule decomposition** is the highest-risk migration step. 58 rules contain overlapping constraints that must be cleanly separated into workflow, agent type, and knowledge layers without gaps.
- **Prompt generator quality** is critical. If generated prompts miss important context, agent output quality drops. Testing must compare generated prompt quality against the current (bloated but comprehensive) approach.
- **Workflow definition format** doesn't exist yet. The schema for defining workflows, steps, agent types per step, and human gates needs to be designed from scratch.
