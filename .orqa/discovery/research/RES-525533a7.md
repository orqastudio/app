# Research: Agent Type Specialization per Workflow Stage

## Task Summary

Research best practices for agent type specialization in multi-agent AI systems, focusing on how leading frameworks handle role definition, role boundaries, knowledge composition, team structure, stage handoffs, configuration formats, and role-based access control.

---

## 1. Framework Comparison: How Multi-Agent Frameworks Handle Agent Specialization

### 1.1 Framework Comparison Table

| Dimension | CrewAI | AutoGen / MS Agent Framework | MetaGPT | ChatDev | LangGraph | CAMEL-AI | Claude Code |
|-----------|--------|------------------------------|---------|---------|-----------|----------|-------------|
| **Role definition** | YAML config: `role`, `goal`, `backstory` per agent | Code-based `AssistantAgent`, `UserProxyAgent` types; YAML in Agent Framework | Python class inheritance from `Role` base; SOP-driven | Prompt-based role-playing (CEO, CTO, Engineer, Tester) | Graph nodes with state; agents as tools or handoff targets | `Role` module with inception prompting | Markdown frontmatter: `role`, `capabilities`, `skills` |
| **Specialization model** | Role descriptions + tool assignment + knowledge sources | Agent types + Magentic manager for dynamic routing | Role classes inherit from base `Role`; actions define capabilities | Fixed organizational hierarchy (CEO > CTO > Programmer > Tester) | Subagents, skills, handoffs, or router pattern | Role-playing pairs with task-specific inception prompts | Universal roles + knowledge injection + skill loading |
| **Knowledge composition** | `knowledge_sources` parameter on agent; embedder config | Semantic Kernel plugins; RAG integration | Shared message pool + memory inheritance | Dialogue-based knowledge sharing | State dict shared across graph nodes | Memory module + tool integrations | Tier 1-3 skill model; knowledge artifacts injected per path |
| **Tool access** | `tools` list per agent; LangChain tools supported | Tool registration per agent; capability-based | Actions define what a role can do | Phase-based tool availability | Tools bound to specific graph nodes | Tool integration per agent | Capability-to-tool mapping per provider context |
| **Team structure** | Crew = team; sequential or hierarchical process | Teams with supervisor/worker or Magentic orchestration | Company-like hierarchy with SOP | Virtual company with departments | Graph topology defines flow | Workforce module for team assembly | Hub-spoke: team lead + background teammates |
| **Communication** | Task-based; delegation between agents | Async messages; event-driven or request/response | Structured documents (not dialogue) | Natural language dialogue | State transitions along graph edges | Role-playing dialogue pairs | Message passing + findings files on disk |
| **Configuration format** | YAML (recommended) + Python code | Python code; YAML in MS Agent Framework | Python classes | JSON config + prompt templates | Python code (graph definition) | Python code | Markdown with YAML frontmatter |
| **Access control** | `allow_delegation` flag; tool scoping | Capability-based; tool registration per agent | Action-level: roles only execute their defined actions | Phase-based: roles active only in their phase | Tool binding per node | Per-agent tool list | Role-to-capability matrix; file-path restrictions |

### 1.2 Framework Deep Dives

#### CrewAI

CrewAI is the most mature YAML-first framework for role-based agent orchestration. Agent definition uses three core attributes: `role` (function/expertise), `goal` (individual objective), and `backstory` (context/personality). Agents compose with resources through three integration points: tools list, knowledge sources, and embedder configuration.

Two execution modes exist:
- **Crews** — autonomous teams where agents have true agency over delegation and approach
- **Flows** — event-driven pipelines for production workloads needing predictability

The hierarchical process mode auto-generates a manager agent that oversees task delegation and reviews outputs, similar to a team lead managing specialists.

**Key insight for OrqaStudio:** CrewAI's YAML-first approach with `role`/`goal`/`backstory` maps naturally to OrqaStudio's existing markdown frontmatter agent definitions. The `knowledge_sources` parameter is analogous to OrqaStudio's knowledge artifact injection.

#### AutoGen / Microsoft Agent Framework

AutoGen (now evolved into Microsoft Agent Framework) provides pre-built agent types (`AssistantAgent`, `UserProxyAgent`, `WebSurferAgent`) plus team collaboration patterns. The Coordinator-Specialist pattern uses a `UserProxyAgent` as project manager that delegates to specialized `AssistantAgents` — the project manager never does specialized work itself.

The Magentic orchestration pattern is notable: a dedicated manager coordinates specialized agents, selecting which agent should act next based on evolving context, task progress, and agent capabilities. This is designed for complex, open-ended tasks requiring dynamic collaboration.

Agents communicate through asynchronous messages supporting both event-driven and request/response patterns. The framework supports pluggable components including custom agents, tools, memory, and models.

**Key insight for OrqaStudio:** The Coordinator-Specialist pattern maps directly to OrqaStudio's orchestrator model. Magentic orchestration is relevant for the discovery stage where investigation paths are dynamic.

#### MetaGPT

MetaGPT's philosophy is "Code = SOP(Team)" — it encodes Standard Operating Procedures into agent prompts. Each role is a Python class inheriting from a base `Role` class, with actions defining capabilities. Roles include Product Manager, Architect, Project Manager, Engineer, and QA Engineer.

The critical differentiator is **structured communication**: agents communicate through documents and diagrams (structured outputs) rather than free-form dialogue. This reduces hallucination and context loss. Each role uses `self._watch([UpstreamMessage])` to subscribe to upstream outputs and `self.set_actions([ActionClass])` to declare capabilities.

Memory inheritance allows summarized information from long-term memory to be inherited by future constraint prompt updates.

**Key insight for OrqaStudio:** MetaGPT's structured communication (artifacts, not dialogue) is exactly what OrqaStudio already does with findings files and artifact graph. The SOP encoding model validates OrqaStudio's rule-based governance approach.

#### ChatDev

ChatDev simulates a virtual software company with organizational roles (CEO, CTO, Programmer, Designer, Tester). Agents cooperate through dialogue in defined phases: Design, Coding, Testing, Documenting.

Each phase activates specific roles — not all roles are active in all phases. This phase-based activation is a form of stage-gated specialization where roles come online as the workflow progresses.

**Key insight for OrqaStudio:** ChatDev's phase-based role activation maps directly to OrqaStudio's discovery/delivery/learning stage model. Only relevant agent types should be available in each stage.

#### LangGraph

LangGraph provides four core multi-agent patterns (from LangChain's architecture guide):

1. **Subagents** — supervisor invokes specialized subagents as tools; stateless with strong context isolation
2. **Skills** — single agent dynamically loads specialized prompts on-demand (progressive disclosure)
3. **Handoffs** — active agent changes via tool calling; state persists across turns
4. **Router** — classifier dispatches to specialists in parallel; typically stateless

The selection framework recommends: subagents for multiple distinct domains + parallel execution; skills for single agent + many specializations; handoffs for sequential workflow + state transitions; router for multiple sources + parallel synthesis.

**Key insight for OrqaStudio:** OrqaStudio combines the subagent pattern (orchestrator spawning specialists) with the skills pattern (progressive knowledge loading). The handoff pattern is relevant for stage transitions.

#### CAMEL-AI

CAMEL uses "inception prompting" to guide role-playing agent pairs. The Workforce module enables team assembly with parallel execution. Core components include Role (initial state), Memory (storage/retrieval), LLMs, and Tools.

**Key insight for OrqaStudio:** CAMEL's inception prompting concept validates the idea that role definition is primarily a prompting concern — the system prompt IS the agent's specialization.

## 2. Specialist vs Generalist: Options Analysis

### 2.1 The Core Tradeoff

| Approach | Advantages | Disadvantages |
|----------|-----------|---------------|
| **Pure specialist** (e.g., "Rust backend implementer") | Deep domain knowledge; focused context window; higher accuracy on domain tasks | Rigid — new domains require new agent types; combinatorial explosion of types; poor at cross-cutting concerns |
| **Pure generalist** (e.g., "implementer") | Flexible — handles any domain; fewer agent types to maintain; natural for cross-boundary work | Context window diluted; lower accuracy on specialized tasks; "jack of all trades, master of none" |
| **Generalist + knowledge injection** (e.g., "implementer with Rust knowledge") | Best of both — role boundary stays clean while domain depth is injected; composable; new domains only need new knowledge, not new agent types | Requires a robust knowledge injection mechanism; knowledge must be well-curated; injection overhead |

### 2.2 Research Evidence

**Google Research (2025-2026):** A controlled evaluation of 180 agent configurations found that "agents focused on specific expertise consistently outperform generalist agents context-switching between roles." On parallelizable tasks, centralized coordination with specialists improved performance by 80.9% over a single generalist. However, on sequential tasks requiring strict reasoning order, every multi-agent variant degraded performance by 39-70%.

**JPMorgan Chase case study:** Proprietary specialist AI agents for investment analysis, risk assessment, and fraud detection — finely tuned to banking regulations — achieved accuracy that generalized systems could not match.

**ChatDev study (2023):** Multi-agent software teams with specialized roles produced working applications in 20 minutes, demonstrating that role specialization enables emergent collaboration.

**Project Sid (2024):** 1,000 agents in Minecraft spontaneously invented specialized roles, suggesting that specialization is a natural emergent property of multi-agent systems.

### 2.3 Recommendation for OrqaStudio

**The "generalist role + injectable knowledge" model is the optimal approach.** This is what OrqaStudio already does with its universal roles (Implementer, Reviewer, Researcher, etc.) composed with knowledge artifacts.

The key refinement from this research is: **agent types should specialize by workflow stage, not by technical domain.** A "frontend implementer" and "backend implementer" differ in which knowledge they load, not in their fundamental role. But a "discovery-stage researcher" and a "delivery-stage implementer" have genuinely different behavioral boundaries (read-only vs read-write, investigation vs execution).

**Proposed taxonomy:**

```
Universal Role (behavioral boundary)
  + Stage Context (workflow stage constraints)
    + Domain Knowledge (injectable expertise)
      = Effective Agent Type
```

Example: `Implementer` (role) + `delivery-stage` (constraints: must write tests, must pass review) + `Rust backend` (knowledge: error handling patterns, Tauri IPC) = what functions as a "Rust backend implementer" without being a separate agent type.

## 3. Knowledge Composition Patterns

### 3.1 How Agent Types Should Compose with Knowledge

Based on research across frameworks, three composition patterns emerge:

#### Pattern A: Compile-Time Binding (MetaGPT, ChatDev)
Knowledge is baked into the role definition. A "Python Engineer" role class has Python knowledge hardcoded in its prompts and actions.

- **Pros:** Simple, predictable, deeply integrated
- **Cons:** Inflexible — new domains require new role classes; knowledge can't be updated independently

#### Pattern B: Runtime Injection (CrewAI, OrqaStudio current)
Knowledge is declared as a dependency and injected when the agent is instantiated. The agent definition says "I need Rust knowledge" and the system provides it at runtime.

- **Pros:** Composable — any knowledge can be paired with any role; knowledge updates don't require role changes
- **Cons:** Requires a robust injection mechanism; knowledge must be self-contained

#### Pattern C: Dependency Injection (PydanticAI)
Typed dependencies are declared on the agent definition and provided at runtime through a DI container. Tools, system prompts, and validators all receive dependencies through a typed `RunContext`.

- **Pros:** Type-safe; testable; clean separation of concerns
- **Cons:** More complex infrastructure; requires a DI framework

### 3.2 Recommended Pattern for OrqaStudio

**Pattern B (Runtime Injection) is the right fit**, which is what OrqaStudio already uses. The refinement from this research:

1. **Agent type definitions declare knowledge requirements** — not specific knowledge artifacts, but knowledge *categories* (e.g., "requires backend-domain knowledge" not "requires KNOW-abc123")
2. **Knowledge plugins satisfy requirements** — a Rust knowledge plugin satisfies the "backend-domain" requirement; a Go knowledge plugin would satisfy the same requirement in a different project
3. **The orchestrator resolves requirements to artifacts** — at delegation time, the orchestrator maps abstract requirements to concrete knowledge artifacts from available plugins

This gives you:
- **Agent types that are portable** across projects (core-framework layer)
- **Knowledge that is pluggable** per technology stack (knowledge-plugin layer)
- **Resolution that is context-aware** (orchestrator matches requirements to available knowledge)

### 3.3 Composition Model

```
Agent Type Definition (core-framework)
  declares: knowledge_requirements: ["backend-domain", "error-handling"]

Knowledge Plugin (e.g., Rust plugin)
  satisfies: ["backend-domain", "error-handling", "async-patterns"]
  provides: KNOW-rust-error-handling, KNOW-rust-async-patterns, ...

Orchestrator (at delegation time)
  resolves: "backend-domain" -> KNOW-rust-error-handling (from Rust plugin)
  injects: resolved knowledge into agent context
```

## 4. Optimal Team Size and Structure

### 4.1 Research Findings

**Google's Agent Scaling Research (2025-2026):**
- Evaluated 180 agent configurations across five architectures
- No universal "more is better" principle — optimal size depends on task structure
- On parallelizable tasks: centralized coordination improved performance 80.9% over single agent
- On sequential tasks: multi-agent degraded performance 39-70%
- Independent multi-agent systems amplified errors by 17.2x
- Centralized systems (hub-spoke) contained error amplification to 4.4x
- Predictive model (R^2 = 0.513) correctly identifies optimal strategy for 87% of unseen task configurations

**Microsoft Azure Architecture Guide (2026):**
- Group chat orchestration should be limited to 3 or fewer agents to maintain effective control
- Maker-checker loops (two agents: one creates, one reviews) are highly effective for quality
- Magentic orchestration (dynamic task ledger) works best for open-ended problems

**Claude Code Agent Teams:**
- Support 2-16 agents per team
- A 3-teammate team uses roughly 3-4x the tokens of a single session
- Teammates spawn within 20-30 seconds, produce results within first minute
- Hub-and-spoke: team lead coordinates, teammates work independently

### 4.2 Communication Patterns Comparison

| Pattern | Structure | Best For | Coordination Overhead | Error Containment |
|---------|-----------|----------|----------------------|-------------------|
| **Hub-spoke** (orchestrator-worker) | Central coordinator delegates to specialists | Most production workloads; clear task decomposition | Moderate — all comms through hub | Good — orchestrator validates |
| **Mesh** (peer-to-peer) | All agents communicate directly | Brainstorming, consensus-building | High — O(n^2) communication | Poor — no central validation |
| **Chain** (sequential pipeline) | Each agent passes to next | Document processing, staged workflows | Low — linear handoffs | Poor — errors cascade |
| **Hierarchical** (nested hub-spoke) | Supervisor delegates to team leads who delegate to workers | Large-scale projects; complex decomposition | Moderate per level | Good — validation at each level |
| **Hybrid** (hub-spoke + selective mesh) | Orchestrator coordinates; workers can peer-communicate when needed | Complex collaborative tasks | Moderate — controlled peer access | Good — orchestrator + peer review |

### 4.3 Persistent vs Ephemeral Agents

| Type | When to Use | Examples |
|------|-------------|---------|
| **Ephemeral** (spawn per task) | Stateless tasks; isolated work units; when context must be fresh | Implementation tasks, code review, research queries |
| **Persistent** (long-running) | Multi-step work requiring accumulated state; ongoing monitoring | Orchestrator, project manager, continuous integration |
| **Hybrid** | Persistent orchestrator managing ephemeral workers | OrqaStudio's current model (persistent orchestrator + ephemeral agent teams) |

The research supports OrqaStudio's current model: a persistent orchestrator that spawns ephemeral agent teams per work batch.

### 4.4 Recommendations for OrqaStudio

1. **Hub-spoke is correct** — OrqaStudio's orchestrator-worker model is validated by research as the most production-ready pattern
2. **Team size: 2-5 agents per batch** — beyond 5, coordination overhead dominates on most tasks
3. **Maker-checker is essential** — the Implementer-Reviewer pair is the most effective quality pattern
4. **Ephemeral workers are correct** — spawning per task and cleaning up avoids stale state accumulation

## 5. Stage Handoff Patterns

### 5.1 How Frameworks Handle Stage Transitions

| Approach | How It Works | Pros | Cons |
|----------|-------------|------|------|
| **Artifact-based handoff** (MetaGPT, OrqaStudio) | Upstream agent produces a structured document; downstream agent reads it | Auditable; no context loss; human-reviewable | Slower — requires file I/O; requires artifact schema |
| **Message-based handoff** (AutoGen, CAMEL) | Agents pass messages directly | Fast; low latency | Context loss in long chains; no audit trail |
| **State-based handoff** (LangGraph) | Shared state dict updated by each agent | Explicit state management; typed | Requires careful state schema design |
| **Tool-based handoff** (LangGraph handoffs) | Agent calls a "handoff tool" that transfers control | Dynamic routing; context-aware | Requires handoff tool implementation per transition |
| **Free-text handoff** | Agents summarize their work in natural language | Flexible; low implementation cost | "Main source of context loss" — treat like a public API |

### 5.2 Quality Gates Between Stages

From the research, quality gates at handoff points are critical:

1. **Pre-gate validation** — before one agent triggers the next, validate outputs include minimum required elements
2. **Structured output enforcement** — constrain model outputs at generation time rather than validating after
3. **Maker-checker loops** — a creation agent and a verification agent iterate until quality criteria are met
4. **Human gates** — certain transitions (e.g., discovery -> delivery) require human approval

### 5.3 Key Metrics for Handoffs

| Metric | Purpose |
|--------|---------|
| Handoff validation success rate | How often handoffs pass quality gate on first attempt |
| Mean repair attempts per stage | How many iterations the maker-checker loop needs |
| Stage-level latency | Time per workflow stage |
| End-to-end cycle time | Total time from input to output |
| Context retention score | How much upstream context survives the handoff |

### 5.4 Recommendations for OrqaStudio

OrqaStudio's artifact-based handoff model is validated as the most robust approach:

1. **Keep artifact-based handoffs** — structured documents (research artifacts, epics, task artifacts) are the handoff mechanism between stages
2. **Add explicit quality gates per transition** — discovery -> delivery requires research artifacts to be complete; delivery -> learning requires acceptance criteria to be verified
3. **Enforce structured output** — agent completion reports must follow a template (What Was Done / What Was NOT Done / Evidence / Smoke Test) — this is already in RULE-af5771e3
4. **Human gates at stage boundaries** — epic `review -> done` already has a human gate; consider human gates at `idea shaped -> promoted` and `research complete -> epic created`

## 6. Agent Type Configuration Format

### 6.1 Format Comparison

| Format | Used By | Pros | Cons |
|--------|---------|------|------|
| **YAML frontmatter in Markdown** | OrqaStudio, Claude Code Skills | Human-readable; version-controllable; supports prose body; progressive disclosure | No native schema validation (requires external tooling) |
| **Pure YAML** | CrewAI, MS Agent Framework | Clean structure; good tooling; CI/CD friendly | No prose body for documentation |
| **JSON** | Some MS tools, ChatDev | Strict schema; universal parser support | Verbose; no comments; poor for long text |
| **Python code** | MetaGPT, AutoGen, LangGraph, CAMEL | Full expressiveness; type checking; inheritance | Not editable by non-developers; harder to audit |

### 6.2 Recommended Configuration Format for OrqaStudio

**Markdown with YAML frontmatter remains the best choice** for OrqaStudio's agent type definitions. This is validated by:

1. CrewAI's success with YAML-first agent definitions
2. Claude Code's Skills standard using SKILL.md with YAML frontmatter
3. The Agent Skills Open Standard using a progressive disclosure model (frontmatter metadata first, full content only when needed)

### 6.3 Recommended Agent Type Schema

Based on the research, here is a recommended schema for OrqaStudio agent type definitions:

```yaml
---
id: AGENT-TYPE-<hash>
type: agent-type
title: "Backend Implementer"
description: "Implements backend features in compiled languages"
status: active

# Which workflow stage(s) this type operates in
stages:
  - delivery

# Base universal role this type specializes
base-role: implementer

# Behavioral boundaries (what this type can and cannot do)
capabilities:
  - file_read
  - file_write
  - file_edit
  - file_search
  - content_search
  - code_search_regex
  - code_search_semantic
  - shell_execute

# Knowledge requirements (abstract categories, not specific artifacts)
# Resolved to concrete artifacts by the orchestrator from available plugins
knowledge-requirements:
  - backend-domain          # Required: language-specific patterns
  - error-handling          # Required: error propagation patterns
  - ipc-patterns            # Optional: if task crosses IPC boundary

# Skills to load (Tier 1 portable skills)
skills:
  - orqa-code-search
  - composability

# Resource profile (for scheduling and parallelism)
resource-profile:
  compilation-risk: high    # May trigger cargo/gcc/etc.
  memory-estimate: heavy    # 300-500MB for Rust compilation
  safe-to-parallelize-with:
    - frontend-implementer
    - researcher
    - writer

# Quality gates this type must pass before completion
completion-gates:
  - lint-clean              # Zero lint warnings
  - tests-pass              # All tests pass
  - reviewer-approved       # Independent reviewer PASS verdict

# Provided by plugin
provided-by: delivery-plugin  # or core-framework
layer: project                # core | project | plugin
---

## Role Description

You are a Backend Implementer specializing in compiled backend languages.
Your responsibility is to implement backend features, commands, and services
following the project's architecture decisions and coding standards.

## Behavioral Boundaries

- You WRITE implementation code (backend source files)
- You RUN build and test commands
- You DO NOT review your own work (delegate to Reviewer)
- You DO NOT modify governance artifacts (delegate to Governance Steward)
- You DO NOT write documentation content (delegate to Writer)

## Required Reading

Before starting any implementation task:
1. Read the task's epic for design context
2. Read the task's acceptance criteria
3. Read coding standards documentation
4. Read architecture decisions relevant to the task area
```

### 6.4 Progressive Disclosure

Following the Agent Skills Open Standard's progressive disclosure model:

- **Level 1 (100 tokens):** `id`, `title`, `description`, `base-role`, `stages` — enough for the orchestrator to select the right type
- **Level 2 (full frontmatter):** `capabilities`, `knowledge-requirements`, `resource-profile`, `completion-gates` — loaded when the agent is being instantiated
- **Level 3 (full document):** Behavioral boundaries, required reading, role description — loaded into the agent's context window

## 7. Role-Based Access Control Patterns

### 7.1 Industry Approaches

**Traditional RBAC limitations for AI agents:**
- RBAC grants permissions to roles, not to specific tasks — problematic since agents often need task-specific access
- An AI agent with excessive permissions can make years worth of mistakes in seconds
- RBAC cannot cope with machine speed — bulk operations happen before alerts fire

**Best practices from research:**
- Give every agent its own low-privilege credentials
- Start each session in read-only mode; grant extra permissions only after explicit elevation
- Route every tool invocation through an authorization service where the policy (not the model) decides
- Segregation of duties: multiple agents with complementary roles provide better security than individual agents with broad permissions

### 7.2 OrqaStudio's Current Model

OrqaStudio already implements a strong RBAC model via RULE-b723ea53 (tool-access-restrictions):

| Capability | Orchestrator | Implementer | Researcher | Planner | Reviewer | Writer | Designer |
|-----------|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| file_read | Y | Y | Y | Y | Y | Y | Y |
| file_edit | Y | Y | -- | -- | -- | Y | Y |
| file_write | Y | Y | -- | -- | -- | Y | Y |
| shell_execute | Y | Y | -- | -- | Y | -- | -- |

This is well-aligned with industry best practices. The key principles:
- **Researcher/Planner are read-only** — investigation cannot mutate state
- **Reviewer can run but not write** — diagnoses but does not fix
- **Writer/Designer can write but not run** — produces artifacts but doesn't execute

### 7.3 Recommendations for Enhancement

1. **Stage-based capability narrowing** — in discovery stage, even Implementer should be read-only (no implementation during discovery)
2. **Path-based restrictions** — Governance Steward can write to `.orqa/` but not to `backend/` or `ui/`
3. **Temporal restrictions** — some capabilities should only be available after prerequisites are met (e.g., shell_execute only after lint passes)

## 8. Recommended Agent Type Taxonomy for OrqaStudio

### 8.1 Three-Layer Plugin Architecture

```
Layer 1: core-framework (universal roles - shipped with OrqaStudio)
  ├── orchestrator        (process coordinator - persistent)
  ├── researcher          (read-only investigation)
  ├── planner             (read-only design)
  ├── implementer         (read-write execution)
  ├── reviewer            (read-only + run verification)
  ├── writer              (read-write documentation)
  └── governance-steward  (read-write .orqa/ only)

Layer 2: stage plugins (workflow-stage specialists)
  ├── discovery-plugin
  │   ├── technical-researcher   (extends: researcher + backend/frontend domain)
  │   ├── product-researcher     (extends: researcher + product/market domain)
  │   ├── ux-researcher          (extends: researcher + ux/design domain)
  │   └── architect              (extends: planner + architecture domain)
  │
  ├── delivery-plugin
  │   ├── frontend-implementer   (extends: implementer + frontend domain)
  │   ├── backend-implementer    (extends: implementer + backend domain)
  │   ├── code-reviewer          (extends: reviewer + code-quality domain)
  │   ├── qa-tester              (extends: reviewer + test-engineering domain)
  │   ├── ux-reviewer            (extends: reviewer + ux-compliance domain)
  │   └── designer               (extends: writer + ux/design domain)
  │
  └── learning-plugin
      ├── lesson-analyst         (extends: researcher + governance domain)
      └── governance-steward     (extends: governance-steward + rule-management domain)

Layer 3: knowledge plugins (domain expertise - injectable)
  ├── rust-knowledge       (satisfies: backend-domain, error-handling, async-patterns)
  ├── svelte-knowledge     (satisfies: frontend-domain, component-patterns, reactivity)
  ├── typescript-knowledge (satisfies: frontend-domain, type-safety, store-patterns)
  ├── tauri-knowledge      (satisfies: ipc-patterns, command-patterns)
  └── ... (extensible per project technology stack)
```

### 8.2 How Extension Works

A stage plugin agent type "extends" a universal role by:
1. **Inheriting the base role's capabilities** — a `frontend-implementer` has all capabilities of `implementer`
2. **Adding stage constraints** — delivery-plugin agents must pass completion gates
3. **Declaring knowledge requirements** — abstract categories that knowledge plugins satisfy
4. **Narrowing behavioral boundaries** — a `frontend-implementer` must not run `cargo` commands

The extension is **not class inheritance** — it's **composition**. The agent type definition declares its base role, and the system composes the effective agent by merging:
- Base role's capabilities and boundaries
- Stage plugin's constraints and gates
- Knowledge plugin's domain expertise

### 8.3 Why This Taxonomy Works

1. **Portable core** — universal roles work on any project type
2. **Pluggable specialization** — stage plugins add workflow-specific types
3. **Injectable expertise** — knowledge plugins add technology-specific depth
4. **No combinatorial explosion** — you don't need "Rust-frontend-implementer" and "Svelte-backend-implementer" as separate types
5. **Clean stage boundaries** — discovery agents are read-only, delivery agents are read-write, learning agents operate on governance artifacts

## 9. Key Takeaways and Design Principles

### 9.1 Validated by Research

These OrqaStudio design decisions are strongly supported by multi-agent framework research:

1. **Hub-spoke orchestration** (orchestrator coordinating specialists) — most production-ready pattern per Google and Microsoft research
2. **Artifact-based handoffs** (structured documents between stages) — MetaGPT's success with structured communication validates this over dialogue-based handoffs
3. **Role-based capability restrictions** — segregation of duties is an industry best practice
4. **Ephemeral workers with persistent orchestrator** — hybrid model avoids stale state while maintaining coordination continuity
5. **YAML frontmatter configuration** — CrewAI and Claude Code Skills standard validate this approach
6. **Generalist roles + knowledge injection** — preferred over specialist role proliferation

### 9.2 New Insights from Research

1. **Stage-based role activation** (from ChatDev) — not all agent types should be available in all stages; constrain what can be spawned per workflow stage
2. **Progressive disclosure for agent definitions** (from Agent Skills Open Standard) — load agent metadata first, full definition only when instantiated
3. **Structured output enforcement at handoffs** (from LangChain best practices) — constrain model outputs at generation time, don't validate after
4. **Maker-checker as the fundamental quality pattern** (from Microsoft Azure guide) — the Implementer-Reviewer pair is the most effective; limit to 3 or fewer agents in a quality loop
5. **Task decomposition is the critical skill** (from Google scaling research) — getting decomposition wrong cascades failures across the entire team; the orchestrator's decomposition ability is the bottleneck
6. **Coordination overhead is real** (from Google scaling research) — multi-agent degrades performance by 39-70% on sequential tasks; only use multi-agent when tasks are genuinely parallelizable

### 9.3 Risks to Monitor

1. **Over-specialization** — too many agent types increases configuration burden and reduces flexibility
2. **Knowledge injection overhead** — injecting too much context dilutes the agent's focus
3. **Handoff context loss** — free-text handoffs are the main source of context loss; enforce structured artifacts
4. **Coordination tax** — on sequential tasks, multi-agent coordination degrades performance; single-agent with skills may be better for sequential work
