## Context

The current migration uses Claude Code's native agent teams (TeamCreate,
Agent, SendMessage, TaskCreate) for hub-spoke orchestration. This works well
for the CLI/connector use case but raises the question: how much of this
coordination logic should live in the app itself?

## The Case for App-Native

- **Provider agnosticism:** Not all LLM providers offer team/agent primitives.
  The app needs multi-agent coordination regardless of which sidecar is active.
- **Deeper control:** Custom orchestration can enforce OrqaStudio-specific
  constraints (role boundaries, artifact scope, review gates, token budgets)
  at the framework level rather than via prompt instructions.
- **Plugin composability:** Agent team definitions could be plugin-provided
  (P1), letting methodology plugins define team structures, role
  specializations, and coordination patterns.
- **Consistent experience:** Same orchestration model whether working via the
  app UI or a connector-generated plugin.

## The Case for Thin Wrappers

- Avoid reimplementing what providers already do well (context management,
  tool routing, parallel execution).
- Maintenance burden of a custom framework vs leveraging evolving provider APIs.
- Connectors already bridge to provider-native primitives — keep the app thin.

## Research References

- **Chroma Context-1** (https://www.trychroma.com/research/context-1) — research
  on context and retrieval for agent systems. Relevant to how agent teams
  share knowledge and coordinate via structured context.

## Libraries to Evaluate

- **CrewAI** (https://crewai.com/open-source) — potentially ideal fit.
  Uses YAML definitions for agents, tasks, and crews which aligns directly
  with P4 (Declarative Over Imperative). Supports role-based agents,
  delegation, and structured task flows. Evaluate whether CrewAI's YAML
  crew/agent/task definitions could map to OrqaStudio's plugin-provided
  role definitions and workflow-driven task routing.
- **AutoGen** (https://github.com/microsoft/autogen) — Microsoft's
  multi-agent conversation framework. Supports customizable agents,
  human-in-the-loop, and group chat patterns. Compare its conversation-based
  coordination model against OrqaStudio's hub-spoke + findings file approach.

## Key Questions

- Where is the boundary between app-level orchestration and sidecar-level
  execution? (The app decides WHAT to do; the sidecar decides HOW to talk
  to the LLM?)
- Can the engine's agent crate provide the orchestration primitives that
  both app and connectors consume?
- What's the minimum viable custom layer that gives provider independence
  without reimplementing inference infrastructure?
- Could CrewAI's YAML definitions be generated by the plugin composition
  pipeline alongside resolved workflows?
- How do these frameworks handle tool access constraints and role boundaries
  compared to OrqaStudio's current enforcement model?
