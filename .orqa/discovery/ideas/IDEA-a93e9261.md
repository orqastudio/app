# App-Level Enforcement Engine

## Principle

The app is the enforcement authority. Connectors are adapters.

```
App (authority)
  → defines rules, hooks, enforcement requirements
  → runs its own enforcement engine for agents working through the app
  → provides the canonical implementation

Connector (adapter)
  → reads the app's rules from installed plugins
  → maps them to the native system's capabilities
  → if the native system can't handle something, the app fills the gap
```

## What This Means

### For Rule Migration

Rules live with their owning plugin regardless of enforcement method. The rule
declares WHAT should be enforced. The app's enforcement engine (Rust) implements
HOW it's enforced when working through the app. Connectors implement HOW it's
enforced in their native system.

A rule does NOT need `requires: connector` — the app enforces it natively.
The connector is a bonus adapter that also enforces it when working outside the app.

### For Connectors

Each connector maps the app's enforcement requirements to its native system:

| App Enforcement | Claude Code Connector | Future OpenAI Connector | Future Cursor Connector |
|----------------|----------------------|------------------------|------------------------|
| PreToolUse validation | PreToolUse hook | API middleware? | Extension hook? |
| PostToolUse validation | PostToolUse hook | ? | ? |
| Prompt injection | UserPromptSubmit hook | System prompt prepend? | ? |
| Session state | Stop hook + SessionStart | ? | ? |
| Knowledge injection | File-write hook | ? | ? |

When adding a new connector:
1. Audit the provider's native hook/enforcement capabilities
2. Map each app enforcement type to a native mechanism where possible
3. Where no native mechanism exists, the app handles it (the agent works through the app)
4. Document the capability mapping in the connector's manifest

### For the App

The app's Rust enforcement engine (`backend/src-tauri/src/`) is the canonical
implementation. It must support everything that any connector implements:

- Rule discovery from installed plugins (already exists via graph scanner)
- Pre/post action validation (needs implementing for in-app agents)
- Prompt/context injection (already exists via system prompt builder)
- Session lifecycle hooks (partially exists via WorkflowTracker)
- Knowledge injection on file operations (needs implementing)

### Priority

The app enforcement engine already exists partially. The gap is that in-app agents
don't go through the same hook pipeline that the Claude Code connector provides.
Closing this gap means:

1. Any rule enforced via Claude Code hooks is ALSO enforced when working through the app
2. New providers (OpenAI, local models) get enforcement for free — the app handles it
3. Connectors only need to implement what they can do BETTER than the app (e.g.,
   Claude Code's native hook system is faster than routing through the app)

## Connector Capability Audit Template

When adding a new connector, fill in this table:

| Enforcement Type | Native Support | Quality | App Fallback Needed? |
|-----------------|---------------|---------|---------------------|
| Pre-action validation | Yes/No | Full/Partial/None | Yes/No |
| Post-action validation | Yes/No | Full/Partial/None | Yes/No |
| Prompt injection | Yes/No | Full/Partial/None | Yes/No |
| Session lifecycle | Yes/No | Full/Partial/None | Yes/No |
| Knowledge injection | Yes/No | Full/Partial/None | Yes/No |
| Thinking mode classification | Yes/No | Full/Partial/None | Yes/No |

This determines the split: native capabilities handled by the connector,
gaps filled by the app.
