## Observation

During the code quality audit (EPIC-e24086ed), the team placed a port allocation reference document in `plugins/core/knowledge/` without also creating a matching documentation page. The user clarified two things:

1. **Plugins are the canonical source of truth** for all content that ships with the product. `plugins/core/`, `plugins/software-kanban/`, `plugins/<name>/` own their docs, knowledge, rules, agents, and skills. The `.orqa/` directory contains **installed copies** (synced by `orqa install`) plus **development-only artifacts** (decisions, lessons, planning, project-specific rules).
2. **Documentation and knowledge ALWAYS come in pairs** — docs = user-facing explanation, knowledge = agent-facing context injection. Creating one without the other is incomplete work.

## Root Cause

No documented architecture decision captured the plugin-canonical model. Agents treated `.orqa/` as the primary authoring location for all content, when in fact `.orqa/` is a mix of installed copies from plugins and dev-only project artifacts. The pairing rule (docs + knowledge together) was implicit in the user's mental model but never captured.

## The Plugin-Canonical Model

Content flows from plugin source directories into `.orqa/` via `orqa install`:

```
plugins/core/knowledge/  ──orqa install──>  .orqa/process/knowledge/
plugins/core/rules/      ──orqa install──>  .orqa/process/rules/
```

Each plugin's `orqa-plugin.json` declares content entries with `source` and `target` paths. The plugin directory is canonical; `.orqa/` is the installed copy.

**How to decide where to write:**
- **Would this ship with `orqa install`?** → Write in the plugin source directory (canonical)
- **Is this project-specific development work?** (decisions, lessons, planning) → Write in `.orqa/` (dev-only)

## The Pairing Rule

Within each location, documentation and knowledge come in pairs:

- **Documentation** (`docs/`) = user-facing. Readable narrative explaining what and why.
- **Knowledge** (`knowledge/`) = agent-facing. Structured content injected into agent context.

## Impact

Without this captured:
- Agents write content directly in `.orqa/` instead of the canonical plugin source, creating drift when `orqa install` runs
- Knowledge artifacts get created without matching docs (agents have context but users can't find explanations)
- Documentation pages get created without matching knowledge (users have docs but agents lack context injection)

## Mitigation

1. Architecture decision [AD-26d8d45d](AD-26d8d45d) captures the plugin-canonical model
2. Documentation placement guide task ([TASK-c3ce6480](TASK-c3ce6480)) will make this discoverable
3. Port allocation task ([TASK-86e8a21b](TASK-86e8a21b)) updated to require paired documentation page
4. Enforce the pairing rule in artifact creation workflows
