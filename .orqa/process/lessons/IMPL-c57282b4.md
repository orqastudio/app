## Observation

A team of 3 agents completed 12 tasks with extensive code changes across the monorepo. None of the code was committed until the session was nearly over. If the session had crashed, all implementation work would have been lost — only the governance artifact files (which were committed earlier) would survive.

## Root Cause

Teammates make code changes but don't commit. The team lead coordinates and verifies but wasn't committing after each task verification. The existing commit discipline rule (RULE-f609242f) says "commit at natural boundaries" but doesn't explicitly address agent team workflows.

## Pattern

After verifying a completed task (reading the findings file, checking compilation):
1. Stage the changed files from that task
2. Commit with a message referencing the task ID
3. Push to local server

This keeps git history aligned with task completions and prevents accumulated risk from uncommitted work.

## Impact

12 tasks worth of changes (~50+ files) sat uncommitted for the entire team session. One crash would have lost everything.
