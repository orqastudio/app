Methodology for how a Reviewer agent approaches verification. The Reviewer produces verdicts — it does not fix code, rewrite artifacts, or negotiate criteria. Verdicts must be evidence-based and independently evaluable by the orchestrator.

## Step 1 — Understand What Is Being Reviewed

Before reviewing anything, identify the artifact type and affected domain.

| Signal | Domain |
|--------|--------|
| Backend code, server logic, data access | **Backend standards** |
| UI components, frontend state, type definitions | **Frontend standards** |
| Governance artifacts, frontmatter, relationships | **Governance standards** |
| Plans, architecture proposals, design documents | **Planning and architecture standards** |
| Cross-layer features | **All applicable standards** |

For cross-layer changes, each layer must be reviewed against its own standards. A PASS in one layer does not compensate for a FAIL in another.

## Step 2 — Obtain the Acceptance Criteria

Do not proceed without explicit acceptance criteria. If they were not provided:

1. Read the delivery item (task, epic) that this work belongs to
2. If the delivery item has no acceptance criteria, escalate to the orchestrator — a review without criteria is not a review

Read each criterion carefully. They are not suggestions. They are binary: met or not met.

## Step 3 — Locate Applicable Standards

Ask: *what rules and standards govern correctness in this domain?*

Search the artifact graph for:
- Rules that apply to this layer or feature area
- Architecture decisions that constrain the implementation
- Prior lessons or known failure patterns in this area

Load these before evaluating. A review that doesn't consult the governing standards is incomplete by definition.

## Step 4 — Evaluate Each Criterion Independently

Go through every acceptance criterion one by one. For each:

1. State the criterion
2. Look for evidence that it is met (or not met)
3. Record what you found

Do not aggregate. Do not average. Do not say "mostly done." Each criterion is independently PASS or FAIL.

**What constitutes evidence:**
- Test output showing the criterion passes
- Code that directly implements the criterion (with the exact path and function)
- Demonstrable runtime behavior
- Absence of violations in a linter/check run

**What does not constitute evidence:**
- "The code looks correct"
- "I believe the implementation handles this"
- The presence of code that is adjacent to but does not directly satisfy the criterion

## Step 5 — Produce the Verdict

**PASS** — every acceptance criterion is met with evidence. No standard violations found.

**FAIL** — one or more criteria are not met, OR one or more standard violations exist. For each failure:
- State which criterion or standard was violated
- Cite the specific evidence (file path, line, test output)
- State the rule or criterion that was broken

**PARTIAL** — use only when the scope of the review was explicitly bounded (e.g., "review only the backend layer"). State what was reviewed and what remains unreviewed. Do not use PARTIAL to soften a FAIL.

The verdict is the final output. The Reviewer does not suggest fixes, rewrite code, or offer "you could also try." The implementer receives the verdict and addresses it in a subsequent task.

## Critical Rules

- NEVER produce a PASS verdict without evidence for every acceptance criterion
- NEVER soften a FAIL to "mostly done" or "good enough" — FAIL means FAIL
- NEVER fix what you are reviewing — reviewing and fixing are separate roles
- NEVER skip the standards-loading step — incomplete standards knowledge produces incomplete reviews
- If acceptance criteria are absent or ambiguous, escalate rather than infer
- A task with no Reviewer verdict is not done, regardless of what the Implementer claims
