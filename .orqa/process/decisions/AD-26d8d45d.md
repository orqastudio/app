## Decision

**Plugins are the canonical source of truth** for everything the app needs to function. The `.orqa/` directory contains **installed copies** (synced via `orqa install`) plus **development-only artifacts** that are not part of the product itself.

Documentation and knowledge artifacts ALWAYS come in pairs within each plugin: docs for users, knowledge for agents.

## Context

During the code quality audit, the team created a port allocation knowledge artifact in `plugins/core/knowledge/` without a matching documentation page. The user clarified two architectural principles:

1. **Plugins are canonical** — `plugins/core/`, `plugins/software-kanban/`, `plugins/<name>/` are the source of truth for all content that ships with the product. The `.orqa/` directory is the *installed* copy, populated by `orqa install`'s content sync mechanism.
2. **Docs and knowledge are always paired** — a knowledge artifact without a matching doc page is incomplete work, and vice versa.

No architecture decision previously captured the plugin-canonical model or the pairing rule. Agents defaulted to ad-hoc placement.

## The Plugin-Canonical Model

### How content flows

```
Plugin source (canonical)  ──orqa install──>  .orqa/ (installed copy)
plugins/core/knowledge/    ──────────────>    .orqa/process/knowledge/
plugins/core/rules/        ──────────────>    .orqa/process/rules/
plugins/core/agents/       ──────────────>    .orqa/process/agents/
```

The `orqa-plugin.json` manifest in each plugin declares `content` entries with `source` and `target` paths. `orqa install` reads these manifests and syncs content from plugin directories into `.orqa/`.

### What lives where

| Location | Role | Content | Examples |
|----------|------|---------|---------|
| **`plugins/core/`** | Canonical source for core framework | Knowledge, rules, agents, docs that ship with every OrqaStudio project | Port allocation, service architecture, app configuration, orchestrator definition |
| **`plugins/<name>/`** | Canonical source for domain plugin | Knowledge, rules, agents, docs specific to that plugin's domain | Svelte patterns (svelte plugin), Tauri patterns (tauri plugin), CLI commands (cli plugin) |
| **`.orqa/process/`** | Installed copies + dev-only | Installed: copies from plugins via `orqa install`. Dev-only: project decisions, lessons, project-specific rules | Installed rules, installed knowledge; also AD-*, IMPL-*, project-specific RULE-* |
| **`.orqa/delivery/`** | Dev-only (never from plugins) | Planning artifacts — epics, tasks, ideas, research, milestones | EPIC-*, TASK-*, IDEA-*, RES-* |
| **`.orqa/documentation/`** | Dev-only (never from plugins) | Development process docs — coding standards, workflow guides, architecture docs | coding-standards.md, commands.md, workflow.md |

### The key distinction

- **Plugin content** = production artifacts that ship with the product. Canonical in plugin directory. Installed to `.orqa/` by `orqa install`.
- **Dev-only content** = project-specific artifacts created during development. Lives only in `.orqa/`. Never comes from a plugin.

### How to tell if something is plugin-canonical or dev-only

1. **Would this content exist in a fresh project that runs `orqa install`?** → Plugin-canonical. Write it in the plugin source directory.
2. **Is this specific to THIS project's development process?** (decisions, lessons, planning) → Dev-only. Write it in `.orqa/`.
3. **Is this a project-specific rule or knowledge that other OrqaStudio projects wouldn't need?** → Dev-only. Write it in `.orqa/process/`.

## The Pairing Rule

Within each plugin (and within `.orqa/` for dev-only content), documentation and knowledge artifacts come in pairs:

- **Documentation** (`docs/` or `documentation/`) — user-facing. Written for humans who want to understand a feature, configuration, or concept. Readable, contextual, narrative.
- **Knowledge** (`knowledge/`) — agent-facing. Written to be injected into agent context when working in a related code area. Structured, concise, action-oriented.

Creating one without the other is incomplete work. The documentation page explains "what and why" for humans; the knowledge artifact provides "what and how" for agents.

## Alternatives Considered

### All docs in .orqa/ (REJECTED)

All documentation in `.orqa/documentation/`. Rejected because:
- Conflates installed content with dev-only content
- Plugin content must be canonical in the plugin so it ships with `orqa install`
- Breaks the plugin distribution model

### No pairing rule — docs and knowledge are independent (REJECTED)

Allow docs without knowledge and knowledge without docs. Rejected because:
- Leads to agent blind spots (knowledge exists but users can't find the explanation)
- Leads to user confusion (docs exist but agents don't get context injection)
- The overhead of creating both is low; the value of consistency is high

## Consequences

### Positive

- Clear canonical ownership — plugin authors own their content, `orqa install` distributes it
- Agents always have context injection when working near documented features
- Users always have human-readable explanations for documented features
- Dev-only artifacts don't pollute the product distribution

### Negative

- Every new document requires creating two artifacts (doc + knowledge)
- Contributors must understand the canonical-vs-installed distinction
- Editing an installed copy in `.orqa/` without updating the plugin source creates drift

### Constraints

- The documentation placement guide (TASK-c3ce6480) must be created to make this decision discoverable
- `orqa install` must be run after plugin content changes to update `.orqa/`
- The three-way diff model (plugin source vs installed baseline vs project copy) detects drift between canonical and installed
