## Rule (NON-NEGOTIABLE)

When working in an agent team, every teammate MUST write their findings to disk BEFORE marking a task complete.

### File Location

```
.state/team/<team-name>/task-<id>.md
```

### What the File Contains

- What was done (files modified, changes made)
- What was NOT done (gaps, deferred items)
- Evidence (command output, verification results)
- Any follow-up tasks identified

### Protocol

1. Teammate completes the work
2. Teammate writes findings to `.state/team/<team-name>/task-<id>.md`
3. Teammate calls `TaskUpdate(completed)`
4. Team lead reads the file to verify completion

### Why

Agent team message delivery is unreliable during concurrent turns. A teammate's completion report can be lost if the team lead is mid-turn. The file on disk is the source of truth — message delivery is a notification, not the evidence.

### Delegation Template

When spawning teammates or assigning tasks, the team lead MUST include:
> Write your findings to `.state/team/<team-name>/task-<id>.md` before marking the task complete.

## Enforcement

A `PostToolUse` hook on `TaskUpdate` checks for the findings file when `status=completed`:

1. **Hook**: `connectors/claude-code/src/hooks/findings-check.ts` — fires on every `TaskUpdate` call
2. **Check**: When `status` is `completed`, scans `.state/team/*/task-<id>.md` for a matching findings file
3. **Action**: If no findings file exists, emits a `systemMessage` warning citing this rule and instructing the agent to write the file before re-marking the task

The hook is non-blocking (warn, not deny) because the task may have been completed outside a team context where findings files are not applicable. The warning ensures agents in team contexts are reminded of the requirement.

## FORBIDDEN

- Marking a task complete without writing the findings file
- Team lead accepting completion without checking the findings file
- Using message delivery as the sole evidence of completion
