## Decision

Tauri `#[tauri::command]` handlers follow a strict three-layer pattern:

1. **Command layer** (`commands/*.rs`) â€” Thin handlers that extract parameters, delegate to domain services, and return IPC-serializable results. Target: 30-50 lines per command function.
2. **Domain layer** (`domain/*.rs`) â€” Focused service modules containing all business logic. Pure functions where possible, explicit dependencies where not. Target: 20-30 lines per function.
3. **Repository layer** (`repo/*.rs`) â€” Data access. SQLite queries, file system operations. No business logic.

The decomposition was driven by `stream_commands.rs` growing to 2,425 lines â€” a single file containing streaming orchestration, tool execution, system prompt building, and session titling. This was extracted into four domain modules: `tool_executor`, `system_prompt`, `stream_loop`, and `session_title`.

## Rationale

Monolithic command files violate the composability principle [PD-af88bb69](PD-af88bb69). When all logic lives in one file:

- Functions can't be reused across commands
- Testing requires spawning the Tauri app context
- Code reviews are unfocused (everything in one diff)
- Parallel agent work causes merge conflicts

The domain service pattern makes each concern independently testable, composable, and maintainable. The command layer becomes a thin adapter between IPC and domain logic.

## Consequences

- Every command file has a maximum complexity budget â€” when it exceeds ~300 lines, domain logic must be extracted
- Domain services are framework-free (no Tauri dependency) and can be unit tested without an app context
- New features add domain services, not command code
- The extraction pattern was applied across the codebase: `setup_commands` â†’ `domain::setup`, `governance_commands` â†’ `domain::governance_analysis`, `artifact_commands` â†’ `domain::artifact_reader`
