# AD-062: Validation Consolidation — libs/validation as Single Source of Truth

## Context

OrqaStudio has 6 separate validation implementations that produce different results depending on which surface you use:

- **app/integrity_engine.rs** — the most comprehensive: 13 checks covering structural integrity, cardinality, cycles, status, delivery paths, parent-child consistency, and body-text references
- **libs/mcp-server/integrity.rs** — a manual copy of the app's implementation, missing 2 checks and already drifting
- **LSP server** — 8 file-level checks implemented independently, covering frontmatter schema and relationship existence but not graph-level properties
- **validate-artifact.mjs hook** — its own JS YAML parser with duplicated relationship checks, no awareness of graph topology
- **Graph visualiser** — computes cluster, orphan, degree, and PageRank metrics client-side in a separate JavaScript Cytoscape pipeline, producing numbers that don't match any server-side metric
- **CLI `orqa enforce`** — calls app integrity engine via IPC when the app is running, has no standalone path

This divergence makes it impossible to get consistent validation results across the dashboard, CLI, MCP, LSP, and hooks. When the dashboard says 3 integrity errors and the CLI says 5, neither is authoritative. When the graph visualiser shows 12 orphaned nodes and the MCP `graph_validate` shows 8, the discrepancy is invisible to the user.

## Decision

### 1. Create libs/validation as the Single Crate

Create `libs/validation` as a shared Rust crate within the workspace. This crate contains ALL validation logic. It is the authoritative implementation.

Starting point: `app/integrity_engine.rs` (most comprehensive). Extract its 13 check modules into `libs/validation/src/checks/`:

- `structural.rs` — broken references, missing inverses, type constraints
- `cardinality.rs` — minimum/maximum relationship counts per type
- `cycles.rs` — cycle detection in relationship graph
- `status.rs` — valid status transitions and terminal state rules
- `delivery.rs` — delivery path completeness (tasks → epics → milestones)
- `parent_child.rs` — parent-child consistency and orphan detection
- `body_refs.rs` — body-text reference validation against known artifact IDs

### 2. Add Graph-Theoretic Metrics

Extend `libs/validation` with graph metrics currently computed only in the JS Cytoscape pipeline:

- Cluster detection (connected components)
- Orphan identification (nodes with degree 0)
- Degree distribution (in-degree, out-degree per node)
- PageRank approximation for artifact importance scoring

These are server-side computations, not client-side analysis. The graph visualiser calls the server for metrics rather than computing them independently.

### 3. Single Output Type

All checks and metrics emit `IntegrityCheck`:

```rust
pub struct IntegrityCheck {
    pub category: CheckCategory,
    pub severity: Severity,
    pub message: String,
    pub artifact_id: Option<String>,
    pub auto_fixable: bool,
}
```

Every consumer receives the same type. No translation layer, no format conversion.

### 4. Consumer Integration

| Consumer | Integration Path |
| ---------- | ----------------- |
| App | Import `libs/validation` as workspace dependency |
| MCP server | Import `libs/validation` as workspace dependency |
| LSP server | Import `libs/validation` as workspace dependency |
| CLI `orqa enforce` | Import `libs/validation` as workspace dependency |
| Hook (validate-artifact.mjs) | Call MCP `graph_validate` tool — no direct Rust dependency |
| Graph visualiser | Call MCP or direct crate import — no independent metric pipeline |

### 5. Delete Duplicates

After each consumer migrates:

- Delete `backend/src-tauri/src/integrity_engine.rs` (source of truth moves to shared lib)
- Delete `libs/mcp-server/src/integrity.rs` (duplicate eliminated)
- Remove JS metric pipeline from graph visualiser

The hook's YAML parsing logic is replaced by delegation to `graph_validate`. No JS reimplementation of validation remains.

## Consequences

- Dashboard, CLI, MCP, LSP, and hooks all produce identical results for the same artifact graph
- Graph health metrics (clusters, orphans, degree, PageRank) computed server-side, surfaced consistently
- Adding a new check requires one change in one place — `libs/validation/src/checks/`
- The MCP `graph_validate` tool becomes the stable interface for any external consumer (hooks, editors, CI)
- `auto_fixable: true` checks can be wired to `orqa enforce --fix` across all surfaces

## Alternatives Considered

1. **Fix the MCP copy by syncing it manually** — rejected: any drift fix is immediately re-introduced at the next change. Two copies will always diverge.
2. **Make the LSP server the single source** — rejected: LSP is file-scoped; it lacks the graph-loading infrastructure needed for cross-artifact checks. The app's integrity engine is better positioned.
3. **Keep JS metrics in the visualiser** — rejected: client-side metric computation means the dashboard and visualiser will always produce different numbers with different algorithmic choices.
4. **FFI from JS hook to Rust** — rejected: the MCP `graph_validate` tool is already a stable interface. Calling it from the hook is the correct path.
