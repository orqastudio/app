## Purpose

The `orqa` CLI is the **single developer interface** for OrqaStudio. There are no separate MCP server or LSP server processes. The CLI binary supports multiple protocol modes, all communicating with the daemon as their backend.

---

## Architecture Model

```text
Developer tools          CLI protocol modes         Backend
─────────────────        ──────────────────         ───────
Claude Code         →    orqa mcp (stdio)      →
VS Code / Cursor    →    orqa lsp (stdio)      →    Daemon (port 10258)
Terminal            →    orqa <command>         →
```text

### Components

| Component | Role | Lifecycle |
| ----------- | ------ | ----------- |
| **Daemon** | App runtime — graph queries, validation, search engine (ONNX + DuckDB) | Long-running. Started by app or `orqa daemon start` |
| **CLI** | Single developer interface with protocol modes | Invoked on demand. Not long-running. |

### Protocol Modes

| Mode | Protocol | Consumer | Transport |
| ------ | ---------- | ---------- | ----------- |
| `orqa mcp` | MCP (stdio) | Claude Code, AI coding assistants | stdin/stdout — spawned by the AI tool |
| `orqa lsp` | LSP (stdio) | IDEs (VS Code, Cursor, Neovim) | stdin/stdout — spawned by the editor |
| `orqa <command>` | Direct CLI | Developer terminal | Human-readable output |

---

## What This Eliminates

- **No separate MCP server process** — `orqa mcp` speaks MCP protocol over stdio
- **No separate LSP server process** — `orqa lsp` speaks LSP protocol over stdio
- **No MCP/LSP ports** — both use stdio, not TCP
- **No lifecycle management** for MCP/LSP — the consumer (editor/AI tool) manages the subprocess

## What Remains

- **Daemon** — single long-running service (port 10258). Graph, search, validation.
- **Search engine** — embedded in the daemon (ONNX + DuckDB), not a separate process
- **CLI** — talks to daemon for all backend operations. Protocol mode determines wire format.

---

## Agent Actions

| Situation | Action |
| ----------- | -------- |
| Adding a new MCP tool | Add it to the CLI's MCP mode handler. The daemon exposes the backend operation. |
| Adding an LSP feature | Add it to the CLI's LSP mode handler. Validation logic lives in the shared validation engine. |
| Adding a new CLI command | Add a direct command. It calls the daemon's internal API. |
| Daemon not running | CLI must detect this and provide a helpful error, not crash. |

---

## FORBIDDEN

- Creating separate MCP server or LSP server processes
- Assigning ports to MCP or LSP (they use stdio)
- Building CLI commands that bypass the daemon for operations the daemon owns (graph, search, validation)
- Embedding long-running state in the CLI (it is invoked on demand, not persistent)
