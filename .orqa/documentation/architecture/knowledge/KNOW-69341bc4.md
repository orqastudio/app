# Glossary

## System Components

| Term | Definition |
| ------ | ----------- |
| **Engine** | Collection of Rust library crates providing all business logic. Not a process — libraries consumed by access layers. |
| **Daemon** | Persistent Rust process: file watchers, MCP/LSP, system tray. Outlives the app. |
| **App** | Tauri desktop app (SvelteKit + Rust). Empty shell without plugins. |
| **CLI** | `orqa` command. Thin Rust wrapper around engine crates. Access method, not business logic. |
| **MCP** | Model Context Protocol. Access protocol exposing engine to LLM tools. NOT an application boundary. |
| **LSP** | Language Server Protocol. Editor/IDE validation against composed schema. NOT an application boundary. |

## Plugin Types

| Term | Definition |
| ------ | ----------- |
| **OrqaStudio Plugin** | Package within OrqaStudio ecosystem. Managed by `orqa install`. |
| **Tool-Native Plugin** | Output artifact generated by a connector (e.g., `.claude/`). NOT an OrqaStudio plugin. |
| **Methodology Plugin** | Defines overarching approach. Provides workflow skeleton with contribution points. One per project. |
| **Workflow Plugin** | Complete sub-workflow for one methodology stage. Owns its state machine. No inheritance. |
| **Domain Knowledge Plugin** | Provides expertise without defining methodology/workflows. No schema recomposition. |
| **Sidecar** | LLM provider integration. App requires at least one. |
| **Connector** | Generates tool-native plugin + watches for changes to regenerate. |
| **Infrastructure Plugin** | Generates enforcement configs (git hooks, linting) from engine rules. |

## Methodology Terms

| Term | Definition |
| ------ | ----------- |
| **Methodology** | Overarching approach (e.g., Agile). Provides named contribution points. |
| **Workflow** | Self-contained sub-workflow for one methodology stage. Owns complete state machine. |
| **Contribution Point** | Named slot in methodology skeleton that a workflow plugin fills. |
| **Resolved Workflow** | YAML file on disk produced by merging plugin contributions. One per stage. Runtime reads only this. |
| **Composed Schema** | Full JSON schema from all plugin-provided artifact type definitions. Used by LSP/MCP. |

## Artifact Terms

| Term | Definition |
| ------ | ----------- |
| **Artifact** | Markdown file with YAML frontmatter + body. Atomic unit of governance data. |
| **Knowledge** | Documentation in agent-consumable chunks (500-2,000 tokens) with injection metadata (tier, roles, paths, tags). |
| **Rule** | Behavioral constraint. **Mechanical** (enforced by tooling) or **advisory** (guidance for agents). |
| **Lesson** | Implementation lesson. If mechanically guardable → becomes mechanical rule immediately. |
| **Principle Decision** | Overarching architecture decision with wide-reaching consequences. Rarely changes. |
| **Planning Decision** | Implementation-level tactical decision. May evolve as implementation progresses. |
| **Guard** | Declarative prerequisite on a state transition. Evaluated mechanically by engine. |
| **Gate** | Human decision point. Five-phase sub-workflow: GATHER, PRESENT, COLLECT, EXECUTE, LEARN. |

## Agent Terms

| Term | Definition |
| ------ | ----------- |
| **Base Role** | One of 8 fixed role definitions. Lives in methodology plugin. |
| **Task-Specific Agent** | Generated at runtime from Base Role + Workflow Context + Domain Knowledge. Ephemeral. |
| **Agent Team** | Hub-spoke model: persistent orchestrator + ephemeral workers. |
| **Token Budget** | Maximum tokens for generated agent's system prompt. Enforced by prompt pipeline. |
| **Injection Tier** | `always` (at spawn), `stage-triggered` (workflow match), `on-demand` (MCP search) |

## Enforcement Terms

| Term | Definition |
| ------ | ----------- |
| **Mechanical Enforcement** | Rules enforced by tooling. Cannot be bypassed by agents. |
| **Advisory Enforcement** | Rules communicated via prompts. Depends on agent compliance. Convert to mechanical where possible. |
| **Target State** | Hand-written artifact representing finished generation pipeline output. Test fixture + validation benchmark. |
