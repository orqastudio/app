# OrqaStudio Core Architecture

## What OrqaStudio Is

A **plugin-composed governance platform** for AI-assisted development. The engine provides Rust library crates. Plugins provide definitions. Nothing is hardcoded in the engine.

**Product Pillars:** Clarity Through Structure | Purpose Through Continuity | Learning Through Reflection

**Core principles:** Accuracy over speed. Mechanical enforcement enables autonomy. The learning loop hardens the system. Zero tech debt.

## Design Principles (P1-P7)

| # | Principle | Constraint |
| --- | ----------- | ------------ |
| P1 | Plugin-Composed Everything | No governance pattern hardcoded in engine. Plugins provide definitions, engine provides capabilities. |
| P2 | One Context Window Per Task | Each agent spawns fresh. No persistent agents, no accumulated context. |
| P3 | Generated, Not Loaded | System prompts generated from plugin registries and workflow state — not loaded from disk. |
| P4 | Declarative Over Imperative | State machines, guards, workflows are YAML declarations validated by JSON Schema — not code. |
| P5 | Token Efficiency as Architecture | 2-4x overhead ratio target. Per-agent prompts: 1,500-4,000 tokens. |
| P6 | Hub-Spoke Orchestration | Persistent orchestrator + ephemeral workers. Every task requires independent Reviewer PASS before acceptance. No autonomous decisions. |
| P7 | Resolved Workflow Is a File | After plugin composition, resolved workflow is a deterministic YAML file on disk. Runtime reads resolved files only. |

## Engine Libraries (Rust Crates)

| Crate | Domain |
| ------- | -------- |
| `graph` | Artifact relationships, traceability, inverse computation |
| `workflow` | State machine evaluation, transition resolution, guard/action execution |
| `prompt` | Five-stage prompt generation from plugin registries |
| `search` | Semantic search over governance artifacts via ONNX embeddings |
| `enforcement` | Rule evaluation, artifact validation, coding standards, linting config generation |
| `plugin` | Installation, composition, schema generation, content management |
| `agent` | Base role definitions, task-specific agent generation |

**MCP and LSP are access protocols, not application boundaries.** Business logic belongs in engine crates.

## Language Boundary

- **Rust** — all libraries, CLI, daemon, connector source
- **TypeScript** — frontend only: SvelteKit UI, `libs/types`, `libs/sdk`, svelte-components, plugin custom views

Connectors are Rust source but generate output in the target tool's native language (e.g., TypeScript/JS for Claude Code).

## Access Layers

Four peer consumers of engine crates:

| Layer | Purpose |
| ------- | --------- |
| **Daemon** | Persistent process: manifest-driven file watchers, MCP/LSP, system tray |
| **App** | SvelteKit + Tauri. Custom UI. Requires sidecar plugins for LLM inference. Enforcement plugin pages dynamically render Run/Fix buttons from the plugin registry — Rust backend dispatches to `orqa enforce --<engine>`. No tools hardcoded in frontend (P1). |
| **CLI** | `orqa` Rust binary — thin wrapper around engine crates. `orqa enforce` dispatches to all installed enforcement engines with dynamically generated flags (`--eslint`, `--clippy`, etc. from installed plugins). `orqa enforce --staged` is what pre-commit calls. |
| **Connector** | Rust source. Generates tool-native plugins (e.g., `.claude/` TypeScript/JS). Daemon manages its file watcher registrations. |

App and connector are peers — both interfaces to the same underlying engine.

## Key Distinctions

- **OrqaStudio Plugin** — managed by `orqa install`, provides definitions to engine
- **Tool-Native Plugin** — output artifact generated by a connector (e.g., Claude Code Plugin). NOT an OrqaStudio plugin.
- **Sidecar** — LLM provider integration plugin. App has no built-in inference — at least one required.
