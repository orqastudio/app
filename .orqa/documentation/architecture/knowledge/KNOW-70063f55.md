# State Machine Design and Enforcement Tooling

## State Machine Rules

- Each workflow plugin owns its **complete** state machine — no inheritance
- State machines are YAML, validated by composed JSON schema
- State categories: `planning`, `active`, `review`, `completed`, `terminal`
- Guards are declarative: field checks, relationship checks, graph queries
- Human gates are five-phase sub-workflows: GATHER, PRESENT, COLLECT, EXECUTE, LEARN

## Two Required Enforcement Layers

An **enforcement plugin** is any plugin that declares an `enforcement` section in its manifest — standalone (e.g., `@orqastudio/plugin-eslint`) or one facet of a broader plugin (e.g., `typescript` provides domain knowledge + tsconfig generator + ESLint contributor rules in one plugin). There are two enforcement sub-types — a plugin can declare one or both:

- **`enforcement-generator`** — owns an enforcement engine: generator script, actions, watcher, config output. Responsible for one output file under `.orqa/configs/`.
- **`enforcement-contributor`** — provides rule files that feed into a generator. Must declare `dependencies` on the generator plugin it contributes to (`orqa install` warns/fails if the generator is not installed).

Every enforcement plugin must provide BOTH:

- **Knowledge injection** — KNOW/DOC artifacts injected into agent prompts so agents learn the standards before writing code. The check should rarely fail if knowledge injection is working.
- **Mechanical checks** — `orqa enforce` as the safety net after agent work and at pre-commit.

**A plugin with check commands but no KNOW artifacts is incomplete.**

## Enforcement Layers (7 Layers)

| Layer | What It Enforces | Mechanism |
| ------- | ----------------- | ----------- |
| **Knowledge injection** | Standards, best practices, common violations, fix patterns | KNOW artifacts from enforcement plugins injected into agent prompts |
| **JSON Schema** (composed) | Artifact structure, required fields, valid types, valid relationships | LSP validation |
| **Git hooks** | Pre-commit checks, relationship validation, artifact format, affected tests | Generated by core plugin; calls `orqa enforce --staged` |
| **Plugin-generated configs** | Any mechanical check from an installed plugin — linting, formatting, type checking, grammar, accessibility, security, license compliance, link checking, and more | Generated to `.orqa/configs/` by generator plugins. Rules in `.orqa/learning/rules/` are the source. |
| **Artifact validation** | Frontmatter correctness, ID format, type-location consistency, relationship validity, knowledge size (500-2000 tokens) | Engine enforcement crate via MCP/LSP |
| **Agent permissions** | File access scoped to role, artifact scope constraints | Generated tool-native plugin permission model |
| **Workflow guards** | Transition prerequisites, gate requirements, AC verification | Engine state machine evaluation |

## `orqa enforce` — Dynamic Dispatch

The universal enforcement command is `orqa enforce`. CLI flags are **dynamically generated from installed plugins** — NOT hardcoded in CLI source. Installing a Vale plugin makes `--vale` valid; uninstalling removes it.

```sh
orqa enforce                  # all registered engines
orqa enforce --staged         # all engines, staged files only (git hooks)
orqa enforce --eslint         # ESLint only
orqa enforce --clippy         # Clippy only
orqa enforce --prettier --fix # Prettier fix mode
orqa enforce --eslint --fix   # ESLint fix mode
```

- Pre-commit hook: `orqa enforce --staged` — does NOT list tool names
- Agent `PostToolUse`/`Stop` hooks: `orqa enforce`
- App UI: Run/Fix buttons on plugin pages dispatch to `orqa enforce --<engine>` / `orqa enforce --<engine> --fix` — dynamically rendered from registry, no tools hardcoded in frontend (P1)
- Adding an enforcement plugin automatically gates it into pre-commit and app UI; no hook or frontend changes

Three surfaces, one pipeline: CLI → git hooks → app UI.

**Authoritative `enforcement-generator` manifest:**

```json
{
  "name": "@orqastudio/plugin-eslint",
  "categories": ["enforcement-generator", "enforcement-contributor"],
  "enforcement": {
    "role": "generator",
    "engine": "eslint",
    "config_output": ".orqa/configs/eslint.config.js",
    "generator": "scripts/generate-config.rs",
    "actions": {
      "check": { "command": "eslint", "args": ["--config", ".orqa/configs/eslint.config.js"] },
      "fix": { "command": "eslint", "args": ["--config", ".orqa/configs/eslint.config.js", "--fix"] }
    },
    "watch": {
      "paths": [".orqa/learning/rules/*.md"],
      "filter": "enforcement_type: mechanical AND engine: eslint",
      "on_change": "regenerate"
    },
    "file_types": ["*.ts", "*.svelte", "*.js"],
    "rules_path": "rules/"
  }
}
```

**Authoritative `enforcement-contributor` manifest:**

```json
{
  "name": "@orqastudio/plugin-typescript",
  "categories": ["domain-knowledge", "enforcement-contributor"],
  "dependencies": ["@orqastudio/plugin-eslint"],
  "enforcement": {
    "role": "contributor",
    "contributes_to": "@orqastudio/plugin-eslint",
    "rules_path": "rules/"
  }
}
```

- `categories` → plural array; valid enforcement values: `enforcement-generator`, `enforcement-contributor`. Multi-area example: `["domain-knowledge", "enforcement-contributor"]`
- `engine` → `orqa enforce --eslint` dynamic flag
- `config_output` → generated config location (always `.orqa/configs/`)
- `generator` → script that composes config from rule files
- `actions` → check/fix commands dispatched by `orqa enforce`
- `watch.paths` + `watch.filter` → daemon registers these; re-runs generator on match
- `file_types` → scoped for `--staged` filtering

## Validation Timing

- **Write time (LSP):** immediate feedback — frontmatter correctness, knowledge size, schema compliance
- **Commit time (pre-commit):** catches anything that slipped through — artifact validation, relationship validity, affected tests

Pre-commit runs tests **scoped to what's changed** — not the full suite.

## Purpose-Specific Guards

Guards should mechanically enforce acceptance criteria, not just check CI:

- A task implementing a function must verify **a test exists AND passes** that defines the function's purpose
- AC enforcement traces from deliverable to verification

## The Learning Loop Hardens Enforcement

```text
Lesson captured → "can we mechanically prevent this?" → if yes: create guard/test/validation rule/plugin-generated config
```text

Every lesson is a candidate for a new mechanical guard. Enforcement tooling grows organically from real failures.

**The more mechanical enforcement exists, the more autonomy agents can have.**
