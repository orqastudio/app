# CLI Architecture and Protocol Modes

## Overview

`orqa` is the **only developer interface** — there are no separate MCP servers, LSP servers, or search processes. The CLI supports three protocol modes, all communicating with a single daemon backend.

## The Daemon

The only long-running service. Maintains in memory:

- **Artifact graph** — all `.orqa/` artifacts and their relationships
- **Search engine** — ONNX Runtime embeddings + DuckDB for semantic search
- **Validation engine** — schema-driven validation shared by all consumers

Starts via the app or `orqa daemon start` for headless/CI use. Listens on port **10100**.

## Three Protocol Modes

| Mode | Command | Protocol | Consumer | Transport |
| ------ | --------- | ---------- | ---------- | ----------- |
| MCP | `orqa mcp` | Model Context Protocol | Claude Code, AI assistants | stdio |
| LSP | `orqa lsp` | Language Server Protocol | VS Code, Cursor, Neovim | stdio |
| Direct | `orqa search`, `orqa graph`, etc. | CLI output | Terminal | Human-readable |

### MCP Mode

Claude Code's `.mcp.json` spawns `orqa mcp` as an stdio subprocess. The CLI translates MCP tool calls into daemon API requests.

Available MCP tools: `graph_query`, `graph_resolve`, `search_semantic`, `search_regex`, `search_research`

Configuration in `.mcp.json`:

```json
{
  "mcpServers": {
    "orqastudio": { "command": "orqa", "args": ["mcp"] }
  }
}
```

### LSP Mode

The editor spawns `orqa lsp` as an stdio subprocess. Provides:

- **Diagnostics** — real-time validation of artifact YAML frontmatter (red squiggles for invalid statuses, wrong relationship types, missing required fields)
- **Completions** — valid status values, relationship types, artifact IDs
- **Hover** — artifact previews and relationship information

### Direct Commands

| Command | Purpose |
| --------- | --------- |
| `orqa search <query>` | Semantic or regex search |
| `orqa graph query` | Query artifact graph |
| `orqa check` | Run validation engine (same checks as LSP) |
| `orqa install` | Sync plugin content to `.orqa/` |
| `orqa enforce` | Universal enforcement entry point |
| `orqa enforce --staged` | Enforcement on staged files only (used by pre-commit) |
| `orqa daemon start/status/stop` | Daemon lifecycle management |

## What This Design Eliminates

| Before | After |
| -------- | ------- |
| MCP server process with own port | `orqa mcp` as stdio subprocess — no port |
| LSP server process with own port | `orqa lsp` as stdio subprocess — no port |
| Port conflict resolution for 3+ services | Single daemon port (10100) |
| PID files and health checks per service | Single daemon with health endpoint |
| Complex startup dependency chains | `orqa daemon start` is the only service |

## Critical Boundary Rule

MCP and LSP are **access protocols** — they are NOT application boundaries. The CLI and daemon contain no business logic. All logic lives in engine crates. The CLI/daemon are thin conduits.
