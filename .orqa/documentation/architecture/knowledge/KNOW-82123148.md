# Target State Specifications

Six target artifacts define what the finished generation pipelines must produce. These are hand-written test fixtures — generation pipelines validate against them. Migration validation is currently in progress (Phase 10).

## Target Status

Targets are read-only during migration. A target is replaced only when the generation pipeline produces matching or superior output. The `targets/` directory is removed after Phase 10 validation is complete.

## A.1 Target Claude Code Plugin (`targets/claude-code-plugin/`)

The ideal `.claude/` output the connector should generate.

**Agent file rules:**

- Agent files use YAML frontmatter (`name`, `description`, `tools`/`disallowedTools`, `model`, `maxTurns`, `skills`)
- No file-level path permissions in agent definitions — Claude Code doesn't support glob path restrictions in agent frontmatter. Use PreToolUse hooks instead.
- Plugin agents cannot set `permissionMode`, `hooks`, or `mcpServers` — silently ignored by Claude Code
- `.claude-plugin/` contains ONLY `plugin.json` — nothing else at that level

**Hook rules:**

- All hooks are thin: `receive event → call daemon/CLI → apply response` — no business logic
- hooks.json generated from plugin declarations (not static)
- Uses `skills/` (current CC standard), not `commands/` (legacy)
- Timeout values in seconds (not milliseconds)

## A.2 Target JSON Schema (`targets/schema.composed.json`)

Full composed schema from all installed plugins. Defines for each artifact type:

- `id_prefix`, `type`, required fields, optional fields, valid statuses, valid relationship types, methodology stage

Artifact types by stage:

- Discovery: discovery-idea, discovery-research, persona, pillar, vision, pivot
- Planning: planning-idea, planning-research, planning-decision, wireframe
- Documentation: doc
- Implementation: epic, task, milestone
- Learning: lesson, knowledge, rule, principle-decision

41 relationship types total (reconciled, deduplicated from all plugins).

## A.3 Target `.orqa/` Structure

Applied directly to live `.orqa/`. Stage-first organization per DOC-fd3edf48 Section 5.1. This IS the live governance data — not a copy in `targets/`.

## A.4 Target Enforcement Configs (`targets/enforcement/`)

What enforcement plugins should generate: githooks (pre-commit, post-commit), eslint configs, clippy.toml, prettierrc, markdownlint.json, tsconfig variants.

Pre-commit checks: frontmatter validation, relationship validation, schema compliance, lint checks, scoped tests, knowledge size constraints (500-2000 tokens), status validity.

## A.5 Target Resolved Workflows (`targets/workflows/`)

One YAML per stage: methodology, discovery, planning, documentation, implementation, review, learning. Each contains full artifact types, state machines, state categories, gates, relationship types, and plugin contribution metadata.

## A.6 Target Plugin Manifests (`targets/plugin-manifests/`)

Corrected `orqa-plugin.json` for all 16 plugins with required fields: `purpose`, `stage_slot` (workflow plugins only), `affects_schema`, `affects_enforcement`, `category`, `uninstallable`, `provides`.

## Validation Rule

Targets are only replaced by generated output — never modified manually to match imperfect generation. If the pipeline doesn't match the target, fix the pipeline.
