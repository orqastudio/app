# State Machine Design and Enforcement Tooling

> This is part of the OrqaStudio Architecture Reference.

---

## 9. State Machine Design (Target State)

- Each workflow plugin owns its complete state machine for its artifact types
- The engine provides the state machine evaluation engine and primitives
- State machines are YAML, validated by the composed JSON schema
- States have categories: `planning`, `active`, `review`, `completed`, `terminal`
- Guards are declarative: field checks, relationship checks, graph queries
- Human gates are five-phase sub-workflows: GATHER, PRESENT, COLLECT, EXECUTE, LEARN
- No inheritance — each workflow plugin's state machine is self-contained
- Relationships between artifacts define flow within and across workflows

## 10. Enforcement Tooling (Target State)

Enforcement operates at two complementary levels: **knowledge injection** and **mechanical checks**. Both are required. A plugin that provides mechanical checks without knowledge artifacts has only half an enforcement strategy.

An **enforcement plugin** is any plugin that declares an `enforcement` section in its manifest. This can be standalone (a dedicated plugin like `@orqastudio/plugin-eslint` whose sole purpose is enforcement) or one facet of a broader plugin (e.g., the `typescript` plugin provides domain knowledge, KNOW/DOC artifacts, a tsconfig generator, AND contributes ESLint rules — all from one plugin). There are two enforcement sub-types — a plugin can declare one or both: `enforcement-generator` (owns the engine, generator, watcher, config output) and `enforcement-contributor` (provides rule data that feeds a generator). An `enforcement-contributor` MUST declare a `dependencies` entry on the generator plugin it contributes to.

- **Knowledge injection** — KNOW artifacts from enforcement plugins are injected into agent prompts before agents write code. Agents learn the standards, common violations, and fix patterns upfront and produce compliant output from the start.
- **Mechanical checks** — `orqa enforce` runs after agent work and at pre-commit. The safety net that catches anything that slipped through. The check should rarely fail if knowledge injection is working.

The check is the safety net. Knowledge is the guidance.

### 10.1 Enforcement Layers

| Layer | What It Enforces | Mechanism |
| ------- | ----------------- | ----------- |
| **Knowledge injection** | Standards, best practices, common violations, fix patterns — taught to agents before they write code | KNOW artifacts from enforcement plugins injected into agent prompts via prompt pipeline |
| **JSON Schema** (composed) | Artifact structure, required fields, valid types, valid relationships | LSP validation against composed schema |
| **Git hooks** | Pre-commit checks, relationship validation, artifact format, affected tests | Generated by core plugin from engine rules; calls `orqa enforce --staged` |
| **Plugin-generated configs** | Any mechanical check declared by an installed plugin — linting, formatting, type checking, grammar, accessibility, security, license compliance, link checking, and more | Generated to `.orqa/configs/` by generator plugins. Rules in `.orqa/learning/rules/` are the source. |
| **Artifact validation** | Frontmatter correctness, ID format, type-location consistency, relationship validity, knowledge size (500-2000 tokens) | Engine enforcement crate, exposed via MCP/LSP |
| **Agent permissions** | File access scoped to role, artifact scope constraints | Generated tool-native plugin permission model |
| **Workflow guards** | Transition prerequisites, gate requirements, AC verification | Engine's state machine evaluation |

### 10.2 `orqa enforce` — Dynamic Dispatch Command

The universal enforcement command is `orqa enforce`. It reads the plugin registry to discover installed enforcement engines and dispatches accordingly. The CLI flags are **dynamically generated from installed plugins** — they are NOT hardcoded in the CLI source.

**Usage:**

```sh
orqa enforce                        # run ALL registered enforcement engines
orqa enforce --staged               # run all engines on staged files (used by git hooks)
orqa enforce --eslint               # run just ESLint engine
orqa enforce --clippy               # run just Clippy
orqa enforce --vale                 # run just Vale (if installed)
orqa enforce --prettier --fix       # run Prettier in fix mode
orqa enforce --eslint --fix         # run ESLint fix mode only
```

**Dynamic flags:** When a Vale grammar plugin is installed, `--vale` automatically becomes a valid flag. When it is uninstalled, the flag disappears. The CLI reads the plugin registry at runtime to build its flag set — adding an enforcement plugin requires no CLI changes.

**How it works:**

- Each generator plugin registers its engine name and commands in its `orqa-plugin.json` manifest
- The manifest declares: engine name (used as the CLI flag), tool, args, file patterns, config path, and whether the engine supports `--fix` mode
- `orqa enforce` reads installed manifests, builds the flag set, dispatches to matching engines
- `orqa enforce --staged` scopes file patterns to staged files before dispatching

**Manifest declaration — authoritative enforcement plugin contract:**

```json
{
  "name": "@orqastudio/plugin-eslint",
  "categories": ["enforcement-generator", "enforcement-contributor"],
  "enforcement": {
    "role": "generator",
    "engine": "eslint",
    "config_output": ".orqa/configs/eslint.config.js",
    "generator": "scripts/generate-config.rs",
    "actions": {
      "check": { "command": "eslint", "args": ["--config", ".orqa/configs/eslint.config.js"] },
      "fix": { "command": "eslint", "args": ["--config", ".orqa/configs/eslint.config.js", "--fix"] }
    },
    "watch": {
      "paths": [".orqa/learning/rules/*.md"],
      "filter": "enforcement_type: mechanical AND engine: eslint",
      "on_change": "regenerate"
    },
    "file_types": ["*.ts", "*.svelte", "*.js"],
    "rules_path": "rules/"
  }
}
```

A contributor plugin declares its sub-type and dependency:

```json
{
  "name": "@orqastudio/plugin-typescript",
  "categories": ["domain-knowledge", "enforcement-contributor"],
  "dependencies": ["@orqastudio/plugin-eslint"],
  "enforcement": {
    "role": "contributor",
    "contributes_to": "@orqastudio/plugin-eslint",
    "rules_path": "rules/"
  }
}
```

A multi-area plugin declares all its categories. The frontend reads `categories` to tag and filter plugins — a plugin appears in every section it declares.

**Key fields:**

- `categories` — array of areas this plugin participates in (`methodology`, `workflow`, `domain-knowledge`, `enforcement-generator`, `enforcement-contributor`, `connector`)
- `engine` — becomes the `orqa enforce --eslint` dynamic flag. The CLI never hardcodes engine names.
- `config_output` — where the generated config is written (always under `.orqa/configs/`)
- `generator` — the script/binary that composes config from rule files in `.orqa/learning/rules/`
- `actions.check` / `actions.fix` — commands registered to the central registry; dispatched by `orqa enforce`
- `watch.paths` — paths the daemon watches for changes
- `watch.filter` — which rule files apply to this engine
- `watch.on_change: "regenerate"` — daemon re-runs the generator on matched file change
- `file_types` — files this engine operates on; used for `--staged` filtering

The daemon reads this at startup/install: registers watch paths, registers `--eslint` as a valid flag, runs the generator for the initial config, and re-runs on watched file change.

**Schema validation:** Declaring `"enforcement-generator"` or `"enforcement-contributor"` in `categories` requires the corresponding `enforcement` block — enforced by JSON schema `if/then` conditionals before installation. A manifest that claims enforcement capability without the configuration block is rejected by `orqa install`. An `enforcement-contributor` without a `dependencies` entry on its generator is also rejected. See DOC-41ccf7c4 section 4.1 for the full category-to-block validation table.

**Git hooks call `orqa enforce`, not specific tools:**

```sh
# pre-commit hook — generated by core plugin
orqa enforce --staged
```

The pre-commit hook does NOT list eslint, clippy, markdownlint, rustfmt, or any other tool by name. Adding a new enforcement plugin automatically gates it into pre-commit. Removing a plugin removes it from pre-commit. No hook changes required.

**Agent lifecycle hooks also call `orqa enforce`:**

- `PostToolUse` and `Stop` hooks call `orqa enforce` to catch issues introduced during agent work
- This is how mechanical enforcement covers agent actions, not just human commits

**The app UI is the third surface — same pipeline:**

Enforcement plugin pages in the app render Run and Fix buttons **dynamically from the installed plugin registry** — no tool is hardcoded in the frontend (P1). The Rust backend dispatches to `orqa enforce --<engine>` or `orqa enforce --<engine> --fix`. Results display inline: errors, warnings, affected files, last run timestamp.

Three surfaces, one pipeline: CLI → git hooks → app UI. All dispatch through `orqa enforce` and all derive their flag set from installed plugin manifests.

**When a plugin declares `enforcement-generator`, it must provide ALL of:**

1. Generator — translates rules into tool-specific config under `.orqa/configs/`
2. `enforcement.engine` declaration — becomes the `orqa enforce --<engine>` CLI flag
3. `check`/`fix` command declarations in manifest
4. File watcher declarations — daemon re-runs generator on rule changes
5. **KNOW artifacts** — agent-optimized knowledge explaining the standards, common violations, and fix patterns; injected into agent prompts so agents produce compliant output before the mechanical check runs
6. **DOC artifacts** — human-readable reference documentation for the standards

**When a plugin declares `enforcement-contributor`, it must provide:**

1. Rule data installed to `.orqa/learning/rules/<domain>/`
2. `contributes_to` — full name of the generator plugin (e.g., `"@orqastudio/plugin-eslint"`)
3. `dependencies` entry on the generator plugin — `orqa install` warns/fails if the generator is not installed
4. **KNOW artifacts** — for the domain it contributes
5. **DOC artifacts** — for the domain it contributes

A plugin with mechanical checks but no KNOW artifacts is incomplete. Agents will fail checks they could have avoided if given the standards upfront.

### 10.3 Validation Timing

Validation is enforced at two levels:

- **Write time (LSP):** immediate feedback while editing — frontmatter correctness, knowledge size constraints, schema compliance
- **Commit time (pre-commit):** catches anything that slipped through — artifact validation, relationship validity, affected tests

Pre-commit runs tests **scoped to what's changed** — not the full test suite, but tests affected by the staged changes.

### 10.4 Purpose-Specific Guards

Workflow guards should mechanically enforce acceptance criteria, not just check for green CI:

- A task that implements a function should verify that **a test exists that defines the function's purpose AND it passes** — not just that all tests pass
- AC enforcement should trace from the task's deliverable to its verification
- The more that can be mechanically verified, the less human review overhead is needed

### 10.5 The Learning Loop Hardens Enforcement

The learning loop should feed directly into enforcement evolution:

1. **Lesson captured:** "this mistake happened"
2. **Analysis:** "can we mechanically prevent this?"
3. **If yes:** create or update a guard, test, validation rule, or plugin-generated config (linting, grammar, security, accessibility, etc.)
4. **Result:** the system hardens itself against past failures over time

Every lesson is a candidate for a new mechanical guard. Avenues for more mechanical enforcement should always be explored through the learning loop. The enforcement tooling grows organically from real failures — this is the Learning Through Reflection pillar in action.
